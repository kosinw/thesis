\begin{abstract}
High-performance processors rely on out-of-order execution, an instruction-scheduling algorithm that is both difficult to implement correctly and prone to timing side-channel attacks without principled defenses.
This thesis develops an approach using the Rocq proof assistant for specifying and proving timing-security properties of hardware circuits.
To express hardware circuits in a proof-friendly style, the thesis introduces \emph{Slate}, a shallowly embedded surface language built on state monads.
Slate provides an imperative-programming model to describe circuits compactly in Gallina.
Since Slate circuits are Gallina functions, reasoning about them in proofs is simple.
The timing-security framework is exercised on a case study of a variable-latency multiplier.
The thesis also presents \emph{Golemma}, a deeply embedded hardware language in Rocq, to establish correctness results for key out-of-order processor submodules.
Golemma provides facilities for modularity and exposes a sequential execution model which makes functional-correctness reasoning simple.
The functional-correctness framework is exercised on a case study of a ring buffer.
Together, these contributions provide a foundation for progression towards integrated proofs of correctness and timing security for full out-of-order processors.
\end{abstract}

\chapter{Introduction} \label{chap:intro}
This thesis contributes an approach for proving correctness and timing properties for hardware circuits described in the Rocq proof assistant.
This work serves as the first step towards the mechanization of entire correctness and timing proofs for a realistic, modern computer system including a single out-of-order core complete with a memory hierarchy.

\section{Background: out-of-order execution} \label{sec:ooo}
Most modern high-speed processors are based on out-of-order (OoO) execution, which allows the CPU to reorder instructions dynamically based on the availability of functional units (FUs) and input operands, minimizing structural hazards.
Instruction-scheduling algorithms supporting out-of-order execution with precise exceptions started with Tomasulo's algorithm~\cite{tomasulo1967}, which first introduced the concept of register renaming to eliminate false dependencies between instructions.
Out-of-order processors combined with another ubiquitous performance optimization, speculative execution, are vulnerable to transient-execution attacks, which in-order processors are not.
Principled defenses must be carefully designed to ensure that attackers cannot exploit programs that are safe for in-order processors to execute but unsafe for out-of-order processors.

\Cref{fig:fig1} shows a block diagram for the particular variant of the out-of-order algorithm discussed in this thesis.
The algorithm is a modified design based on the MIPS R10000 (R10K) processor~\cite{r10k1996}.

\begin{figure}[H]
  \centering
  \includegraphics[width=0.75\textwidth]{figures/figure1.png}
  \caption[Out-of-order machine diagram]{Diagram of an out-of-order machine based on the R10K processor. Arrows signify communication between submodules.}
  \label{fig:fig1}
\end{figure}

Our algorithm is split into seven separate pipeline stages: fetch, decode, rename, issue, execute, writeback, and retire.
The frontend stage fetches instructions from the instruction stream and decodes them into an internal data type the machine can schedule.
The renaming stage then maps architectural registers to a larger physical register file so false dependencies can be dissolved.
The issue stage selects ready renamed operations and sends them to functional units that execute the actual computation.
The writeback stage then posts results to the physical registers so that later instructions can consume operands as they become ready.
The retire stage retires instructions in program order, committing their effects to the architectural state.

Our design diverges from the R10K algorithm in two major aspects to simplify which invariants are needed to write proofs.
The most important difference is how the renaming mechanism works.
The original R10K algorithm records, in the reorder buffer (ROB), the previous physical register index for each instruction's destination so that the rename map can be rolled back precisely when a younger path is squashed.
We instead maintain two register alias tables (RATs), avoiding the need to track a per-instruction previous physical destination in the ROB.
The future table holds the most up-to-date speculative mapping from architectural registers to physical registers, while the commit table records the mapping as of the last retired instruction.
Squashing speculative state can then be performed by reverting the future back to the commit table.

Rolling back the alias tables alone would be insufficient, because speculative execution can still overwrite physical register contents before those instructions retire; restoring only the architectural-to-physical mapping would leave stale or partially updated values visible under the recovered names.
Our design therefore pairs the dual RAT with explicit snapshots of register values (not just of the rename map).
Enough snapshots are retained to reconstruct the physical register file as it stood at the last retired instruction, so a squash restores both the mapping from each architectural register to its physical backing and the bit patterns stored in those physical registers at that point.
The maximum number of snapshots is finite, so it bounds the number of branches the machine can speculate ahead; once that budget is exhausted, the issue stage must stall until retire frees snapshot capacity.

A second major deviation concerns speculative memory accesses.
To make timing properties easier to state and prove at first, we delay all speculative loads and stores.
This policy is extremely pessimistic: it sacrifices the performance that comes from speculative execution when interacting with memory and is not meant as a proposal for a serious, production-ready design.
Looking ahead, we expect that a realistic design can be achieved by pairing speculative memory execution with a dynamic taint-tracking defense, together with a mechanized proof that the resulting design still satisfies the same class of timing-security obligations; that extension is left to future work.

\section{Goal: verifying hardware circuits}
This thesis develops techniques for proving correctness and timing properties of synthesizable hardware described in Rocq.
The intended workflow is to state and prove theorems against source-level circuit semantics, then relate them to the semantics of synthesizable Verilog through a verified compiler.
Prior work on security proofs for hardware tends to reason about simplified ISA-level models instead of RTL.
This thesis focuses on synthesizable circuits because they are consumable by downstream synthesis tools for future evaluation on FPGAs or ASICs.

\chapter{Related work} \label{chap:related}

This chapter first surveys prior work on proving functional correctness for hardware circuits (\Cref{sec:related-correctness}).
Then it covers prior work on timing security, both the attacks that motivate this work and the hardware and software defenses that mitigate them (\Cref{sec:related-timing}).

\section{Correctness} \label{sec:related-correctness}

Kami~\cite{choi2017} is a hardware-description language, embedded in Rocq, for specifying and verifying hardware circuits in the style of Bluespec SystemVerilog.
Circuits are semantically treated as labeled transition systems composed of guarded atomic rules, and correctness arguments proceed by refinement between more-abstract and more-concrete specifications.

Ko\^{i}ka~\cite{koika2020} is another language for rule-based hardware design that gives Bluespec-style rules a precise sequential semantics characterized by the \emph{one-rule-at-a-time} (ORAAT) property.
Circuits are described as collections of rules over register state, and the language provides a scheduler that resolves rule conflicts while preserving a one-rule-at-a-time execution model.
Ko\^{i}ka is accompanied by a verified compiler that lowers circuit descriptions to RTL, providing an end-to-end path from high-level specifications to synthesizable netlists.
However, Ko\^{i}ka does not support a module system, so circuits must be described monolithically, and it lacks support for compositional reasoning.

\section{Security} \label{sec:related-timing}

The Meltdown~\cite{lipp2018} and Spectre~\cite{kocher2019} attacks, disclosed in 2018, demonstrated that speculative execution, while critical for performance, introduces new side-channel vulnerabilities absent in in-order processors.
Both attacks exploit the fact that transiently executed instructions leave measurable traces in microarchitectural state, such as cache-residency patterns, that an attacker can recover.
These attacks motivated a large body of research~\cite{koruyeh2018,maisuradze2018,kiriansky2018,schwarz2019,minkin2019} on transient-execution attacks.

Hardware-software contracts~\cite{guarnieri2021} formalize hardware defenses by specifying, for a particular design, both what information is leaked and which speculative paths are explored.
An analysis of an abstract out-of-order processor protected by speculative taint tracking~\cite{yu2019} shows that the defense satisfies a contract in which leakage events are branch targets, addresses of loads and stores, and operands of variable-latency instructions.
The key limitation of this line of work is that it reasons about abstract processor models rather than concrete designs expressible in a hardware-description language; this thesis targets the gap to synthesizable RTL.

At the software level, prior work on constant-time verification for compilers~\cite{conoly2025} developed techniques for proving constant-time properties for nondeterministic programs.
Using a formulation of constant time as a single-copy property, the work proves end-to-end preservation of constant-time routines compiled from Bedrock2~\cite{erbsen2021} source to RISC-V machine code.
The single-copy formulation of constant time used in this thesis is taken from that work, repurposing the approach from software compilers to hardware verification.

SecSep~\cite{song2025} addresses the gap between source-level constant-time guarantees and speculative execution on real hardware through an assembly-level transformation framework.
SecSep targets processors similar to ProSpeCT~\cite{daniel2023}, which partitions memory into regions whose speculative loads are delayed until non-speculative and regions whose loads are not, provably enforcing the constant-time policy without relying on fencing.
SecSep rewrites compiled assembly programs so that secret and public data are partitioned on the stack, using a new typed assembly language called \emph{Octal} to guide the transformation.
The approach correctly handles secrets on the stack and achieves a low additional runtime overhead on cryptographic programs.

Prior work on hardware timing security~\cite{duxovni2018} builds off the Kami framework to reason about timing-security properties of hardware circuits, showing that the same embedding used to verify functional correctness can also support security arguments.
More recent work~\cite{lau2024} formally specifies and verifies strong timing isolation for hardware enclaves, reasoning about cycle-accurate circuit semantics with respect to an air-gapped-machine formulation of isolation and applying the method to a multicore, pipelined RISC-V design.
This thesis continues in the same tradition, carrying these timing-security proof techniques into the setting of individual submodules of a large out-of-order machine.

\chapter{Verifying timing security for circuits} \label{chap:Slate}

This chapter presents a framework for timing-security verification of hardware circuits.
It begins by giving a formalism of state machines (\Cref{sec:timing-state-machine}) to model hardware circuits and then introduces \emph{Slate}, a shallowly embedded language in Rocq for specifying hardware circuits (\Cref{sec:timing-monads}).
Finally, it defines hardware timing security and develops the simulation-based proof structure used to establish it (\Cref{sec:timing-security}).

\section{State machines} \label{sec:timing-state-machine}

A \emph{state machine} is an abstraction used to describe the behavior of a computation by defining its possible states and the rules for transitioning between them.
Hardware modules function as physical state machines.
Registers serve as the memory that holds the current state, while combinational logic describes the transition function that determines both the module's outputs and its next state as a function of the current state.
We view hardware modules as interacting through well-defined interfaces composed of methods rather than raw, unstructured signals.
Each discrete state transition in the system corresponds to the execution of a method, supplied with arguments that represent the transition's data.
This approach is consistent with prior work~\cite{choi2017,koika2020}, which views hardware modules as high-level transition systems whose transitions lie on external method boundaries.
We adopt the following formal definition of a state machine.

\begin{definition}[State machine]
  A state machine $\mathcal{M}$ is a 5-tuple $(S, s_0, I, O, \delta)$ where $S$ is a set of states, $s_0 \in S$ is the initial state, $I$ is a set of inputs, $O$ is a set of outputs, and
  \[
    \delta : S \times I \rightharpoonup S \times O
  \]
  is a partial transition function.
  Here $\xrightharpoonup{}$ denotes that certain methods can abort and produce no state change or output.
\end{definition}

We model hardware circuits as such machines.
A state $s \in S$ captures the current values of the sequential logic relevant to the module.
Each input in $I$ represents a \emph{method invocation}: a variant type carrying a method tag together with the argument values for that call.
The transition function $\delta$ is partial on $S \times I$ to reflect that the model may not produce a well-defined successor state and output when the call is \emph{aborted} by a guard (for example, dequeuing from an empty buffer).
An aborted call leaves the circuit in the same configuration.
This convention simplifies trace definitions and simulation proofs: each input sequence determines one run, and the optional output indicates whether the call produced a visible event in the output trace.

For the purpose of reasoning about security with this framework, we additionally restrict attention to circuits whose behavior is \emph{deterministic} in the sense above:
\[
  \begin{aligned}
  &\forall s \in S.\; \forall \iota \in I.\;
    \forall s_1,s_2 \in S.\; \forall o_1,o_2 \in O.\\
  &\qquad
    \delta(s,\iota) = (s_1,o_1)
    \land
    \delta(s,\iota) = (s_2,o_2)\\
  &\qquad\implies
    s_1 = s_2 \land o_1 = o_2.
  \end{aligned}
\]
In words, if a state and input can step to two results, then those results must be identical.
This restriction is convenient for reasoning about the proof obligations in \Cref{sec:timing-security}, where we compare runs under related inputs and want equality of traces without existentially quantifying over implementation-defined resolution of nondeterminism in proofs.
Determinism is a design choice of convenience, not necessity; as discussed in \Cref{chap:Golemma}, the framework for deeply embedded circuits can use nondeterministic semantics (in particular, omnisemantics~\cite{chargueraud2023}) to describe specifications.

Given an input sequence $\iota_1, \iota_2, \ldots, \iota_n \in I$, repeatedly applying $\delta$ from the initial state $s_0$ defines a \emph{run} of the machine.
The corresponding sequence of per-step observations is called the machine's \emph{observation trace}.
Concretely, the machine $\mathcal{M}$ produces a sequence of optional outputs by repeatedly applying $\delta$ from $s_0$ for each input $\iota_k \in I$:
\[
  s_0 \xrightarrow{\iota_1 / o_1} s_1 \xrightarrow{\iota_2 / o_2} s_2 \;\cdots\; \xrightarrow{\iota_n / o_n} s_n, \qquad o_i \in O,\ s_i \in S.
\]
The \emph{observation trace} of the run is the list:
\[
  \mathcal{T}\bigl( \mathcal{M}, \vec\iota \bigr) \;\triangleq\; \bigl(o_1,o_2,\ldots,o_n \bigr)
.\]
For timing security, an attacker interprets elements of $O$ (modulo erasure, described in \Cref{sec:timing-security}) as leaked information.
Aborted transitions do not produce outputs in the observation trace.

\section{Encoding circuits with Slate} \label{sec:timing-monads}

This section introduces Slate, a language for describing hardware as executable descriptions in Rocq.
It first compares shallow and deep embeddings and explains why Slate is implemented as a shallow embedding (\Cref{sec:timing-monads-shallow}).
It then discusses guarded state monads as a way to model imperative programs shallowly in Rocq (\Cref{sec:timing-monads-state}).
Finally, it presents a small illustrative example of using the Slate language (\Cref{sec:timing-monads-slate}).

\subsection{Shallow and deep embeddings} \label{sec:timing-monads-shallow}
A \emph{shallow embedding} encodes a source language directly as terms of the host language.
In this style, the host's type system, reduction rules, and terms \emph{are} the semantics of the object language.
In Rocq, shallow embeddings are represented as ordinary Gallina functions rather than abstract syntax trees requiring a separate interpreter.
By contrast, a \emph{deep embedding} defines explicit syntax via inductive datatypes for commands and expressions.
Since syntax trees do not execute by reducing as Gallina terms, deep embeddings must come with separate semantics, such as an interpreter or operational semantics.

Each style has distinct advantages and costs.
Shallow embeddings are typically easier for verification work, because they directly leverage Rocq's facilities for Gallina; many proof steps reduce by unfolding and simplification rather than by reasoning about an interpreter or operational semantics.
The main drawback is that source-level program transformations, such as compiler passes, are tricky to implement, since syntax is not explicit.
These tasks often require reification and metaprogramming support to recover a syntax tree.

Deep embeddings make syntax explicit from the beginning, so compiler construction and optimization passes are easier to express.
However, they impose additional verification overhead: one must define and reason about an interpreter (or step relation), and prove that this semantics aligns with the intended meaning.
As a result, proofs on deep embeddings are more-involved than similar theorems on shallow embeddings.
\emph{Slate}, the surface language used for timing-security verification in this thesis, is a shallow embedding.
Its core semantics work as an imperative language with mutable references.
We introduce the state-monad structure that underlies Slate method definitions.

\subsection{State monads and actions} \label{sec:timing-monads-state}
\paragraph{State monads}
For a fixed state type $s$, the \emph{state monad} is the type family that maps each result type $t$ to
\[
  \mathsf{State}_s(t) \;\triangleq\; s \to (s * t)
,\]
where an element of $\mathsf{State}_s(t)$ is a stateful computation that takes an input state and returns a new state together with a result.
This representation is powerful because computations can be sequenced with the monadic bind operation.
Mutable references do not need to be primitively supported by the metalanguage (i.e., Gallina); instead, stateful programs are ordinary functions that transform an explicit state argument.
As a consequence, one can describe sequential hardware behavior in a pure functional setting and still recover the usual intuition of sequential logic, by reading registers at the start of the cycle, performing some combinational function on those values, and updating the registers at the end of the cycle.

\Cref{fig:state-monad-core} shows the state monad encoded in Rocq together with its basic operations.
The constructor $\mathit{ret}$ lifts a pure value into the monad without changing the state, while $\mathit{bind}$ sequences two computations by feeding the result and resulting state of the first computation into the second.
The infix operator \mintinline{coq}{>>=} is another way to express $\mathit{bind}$.
State monads also provide state-specific primitives, notably $\mathit{get}$ and $\mathit{put}$: $\mathit{get}$ returns the current state as a value, and $\mathit{put}$ replaces the current state with a new one.

\begin{figure}[H]
\begin{minted}{coq}
Definition State s t := s -> (s * t).

Definition ret {s t} (x : t) : State s t :=
  fun st => (st, x).

Definition bind {s t1 t2} (m : State s t1) (f : t1 -> State s t2) :=
  fun st =>
    let '(st', x) := m st in
    f x st'.

Definition get {s} : State s s :=
  fun st => (st, st).

Definition put {s} (x : s) : State s unit :=
  fun _ => (x, tt).
\end{minted}
\caption[State-monad definitions]{State-monad definitions in Rocq.}
\label{fig:state-monad-core}
\end{figure}

\paragraph{Guarded state monads}
% For a fixed state type $s$, the \emph{state monad} is the type family that maps each result type $t$ to
For a fixed state type $s$, the \emph{guarded state monad} or \emph{action} is type family that maps each result type $t$ to
\[
  \mathsf{Action}_s\;(t) \;\triangleq\; s \to \mathit{option}\;(s * t)
,\]
that is, a function that takes an input state and either returns a new state together with a result or aborts.
The framework uses actions to model method calls that are enabled only when their guards hold.
The $\mathit{Some}$ case represents a successful step that returns an updated state and a value.
The $\mathit{None}$ case represents an aborted step (for example, when a guard is false).
\Cref{fig:guarded-state-monad-core} shows the guarded state monad encoded in Rocq together with its basic operations.
$\mathit{ret}$, $\mathit{bind}$, $\mathit{get}$, and $\mathit{put}$ work similarly to their state-monad analogues.

\begin{figure}[H]
\begin{minted}{coq}
Definition Action s t := s -> option (s * t).

Definition ret {s t} (x : t) : Action s t :=
  fun st => Some (st, x).

Definition bind {s t1 t2} (m : Action s t1) (f : t1 -> Action s t2) :=
  fun r =>
    match m r with
    | Some (s, x) => f x s
    | None => None
    end.

Definition get {s} : Action s s :=
  fun st => Some (st, st).

Definition put {s} (x : s) : Action s unit :=
  fun _ => Some (x, tt).

Definition modify {s} (f : s -> s) : Action s unit :=
  fun st => Some (f st, tt).

Definition fail {s t} : Action s t :=
  fun _ => None.
\end{minted}
\caption[Guarded-state-monad definitions]{Guarded-state-monad definitions in Rocq.}
\label{fig:guarded-state-monad-core}
\end{figure}

\paragraph{Modify}
$\mathit{modify}$ updates the state by applying a function $\mathit{f}$.
It can be used as a derived write primitive: read the current state, transform it, then commit the result.
In hardware, this operation is a natural fit for register updates where the next state is expressed as a pure function of the current state.
Proof obligations involving $\mathit{modify}$ usually reduce to equalities between functions applied to the same input state.

\paragraph{Guard failure}
$\mathit{fail}$ returns $\mathit{None}$ for any input state.
In hardware-description semantics, a $\mathit{None}$ result represents an aborted method call (for example, violating a ready condition).
At the state-machine level, guard failure is normalized into a step with no observable output and no state changes.
Introducing failure in the monad itself keeps reasoning about guards local to method definitions.

\paragraph{Submodule calls}
The operator $\mathit{call}$ implements modular method calls over nested state.
It uses $\mathit{getter}$ to project a substate, runs an $\mathit{Action}$ on that substate, and then reintegrates the updated substate using $\mathit{setter}$ as shown in \Cref{fig:action-call}.
If the callee action fails, the caller fails.
Submodule calls with guarded state monads calling semantics in Bluespec SystemVerilog, where a method call that fails to fire causes the caller to fail to fire.

\begin{figure}[H]
\begin{minted}{coq}
Definition call {s a b}
  (getter : s -> a)
  (setter : a -> s -> s)
  (action : Action a b) : Action s b :=
  fun st =>
    fun _ =>
      match action (getter st) with
      | Some (st', x) => Some (setter st' st, x)
      | None => None
      end.
\end{minted}
\caption[Call operator for actions]{$\mathit{call}$ operator lifts the effects of a submodule action to a larger state.}
\label{fig:action-call}
\end{figure}

\subsection{Syntax} \label{sec:timing-monads-slate}
The syntax for Slate is implemented using the extensible parser in Rocq.
Crucially, the parser elaborates Slate syntax directly into actions as described above; it does not introduce a syntax tree.
Consider \Cref{fig:slate-surface}, which implements the $\mathit{step}$ method of a GCD module.
The state type $\mathit{GCDState}$ holds two registers $\mathit{x}$ and $\mathit{y}$.
The method reads both registers into let bindings, aborts immediately when \mintinline{coq}{y = 0} (the computation is already finished), and otherwise subtracts the smaller value from the larger.
This style reads like imperative code while denoting a Gallina term.

\begin{figure}[H]
\begin{minted}{coq}
Record GCDState := { x : Bit#(16); y : Bit#(16) }.

Definition step : Action GCDState unit :=
  begin
    let xv = `x`read();
    let yv = `y`read();
    when (yv == 0)
      abort;
    if (xv >= yv) begin
      `x`write(xv - yv)
    end else begin
      `y`write(yv - xv)
    end
  end.
\end{minted}
\caption[GCD step, Slate]{The $\mathit{step}$ method of a GCD module using Slate.}
\label{fig:slate-surface}
\end{figure}

The same method can be written directly with monadic operators (\Cref{fig:slate-bind}).
This version makes sequencing explicit through \mintinline{coq}{>>=}, but is noticeably less readable for method bodies of this length.
In practice, both forms are definitionally equal after notation elaboration.

\begin{figure}[H]
\begin{minted}{coq}
Definition step : Action GCDState unit :=
  get >>= fun st =>
  (if st.(y) =? 0 then fail else ret tt) >>= fun _ =>
  if st.(x) >=? st.(y) then
    modify (fun s => {| x := s.(x) - s.(y); y := s.(y) |}) >>= fun _ =>
    ret tt
  else
    modify (fun s => {| x := s.(x); y := s.(y) - s.(x) |}) >>= fun _ =>
    ret tt.
\end{minted}
\caption[GCD step, monadic notation]{The $\mathit{step}$ method of a GCD module using inline monadic notation.}
\label{fig:slate-bind}
\end{figure}

\section{Specifying and proving timing security} \label{sec:timing-security}

Establishing that a hardware circuit is free from timing side channels requires both a precise definition of what it means to be secure and a methodology for proving that a given circuit satisfies that definition.
This section addresses both concerns.
It first introduces cryptographic constant time (CCT) in the software setting and then extends the idea to hardware circuits as hardware constant time (HCT), stated in terms of erasure contracts that model the attacker's observational capabilities (\Cref{sec:cct,sec:hct}).
It then develops the simulation-based proof strategy used to establish HCT: using lockstep simulation with trace-equivalence lemmas that connects simulations to the HCT definition (\Cref{sec:timing-proof-lockstep}).

\subsection{Cryptographic constant time} \label{sec:cct}
Cryptographic software often cares about the \emph{cryptographic-constant-time} (CCT) property: attackers should not observe leakage events such as branch direction, memory-access patterns, or variable-latency instructions influenced by secret inputs.
This principle has been formalized and mechanically verified for software programs in multiple prior works~\cite{guarnieri2021,conoly2025,almeida2016}.
\Cref{fig:cct-intuition} illustrates the key intuition with two small routines: one has a secret-dependent branch and is not constant time, while the other computes through bit masking and avoids control-flow dependent on secret data.

More formally, constant-time security is stated as an observational equality under varying secrets: if two executions agree on public inputs, then they must be indistinguishable with respect to the attacker-observable trace.
The next definition makes constant time precise in the traditional two-copy (hyperproperty) style~\cite{clarkson2010}, before introducing an equivalent one-copy formulation.

\begin{figure}[H]
\begin{minipage}[t]{0.48\linewidth}
\begin{minted}{c}
// Not constant-time
uint32_t f_bad(x, y, secret) {
  uint32_t x;
  uint32_t y;
  uint8_t secret;

  if (secret) return x;
  else return y;
}
\end{minted}
\end{minipage}\hfill
\begin{minipage}[t]{0.48\linewidth}
\begin{minted}{c}
// Constant-time
uint32_t f_ct(x, y, secret) {
  uint32_t x;
  uint32_t y;
  uint8_t secret;

  uint32_t m = 0u - (secret & 1u);
  return (x & m) | (y & ~m);
}
\end{minted}
\end{minipage}
\caption[Constant-time example]{Constant-time example: secret-dependent branch (left) versus branch-free masked selection (right).}
\label{fig:cct-intuition}
\end{figure}

\begin{definition}[Cryptographic constant time as a two-copy hyperproperty] \label{def:two-copy}
A state machine $\mathcal{M} = (S, s_0, I, O, \delta)$ is \emph{cryptographic-constant-time} (CCT)
if given partitions
\[
  s = (s^{\mathrm{pub}}, s^{\mathrm{sec}}), \quad
  i = (i^{\mathrm{pub}}, i^{\mathrm{sec}}), \quad
  o = (o^{\mathrm{pub}}, o^{\mathrm{sec}})
\]
of states, inputs, and outputs into public and secret components, such that for any two
states $s_1, s_2 \in S$ with $s_1^{\mathrm{pub}} = s_2^{\mathrm{pub}}$, and for any two
input sequences $(i_1, i_2, \ldots)$ and $(i_1', i_2', \ldots)$ in $I$ with $i_k^{\mathrm{pub}} = i_k^{\prime\,\mathrm{pub}},$
the public output components agree at every step $k$, $o_k^{\mathrm{pub}} = o_k^{\prime\,\mathrm{pub}}.$
\end{definition}

This definition is relational: it compares two universally quantified executions and is therefore a hyperproperty.
Next, we introduce a one-copy formalism of constant time which is logically equivalent to \Cref{def:two-copy}.

\begin{definition}[Cryptographic constant time as a one-copy property]
A state machine $\mathcal{M} = (S, s_0, I, O, \delta)$ is \emph{cryptographic-constant-time} (CCT)
if given partitions of states, inputs, and outputs
\[
  s = (s^{\mathrm{pub}}, s^{\mathrm{sec}}), \quad
  i = (i^{\mathrm{pub}}, i^{\mathrm{sec}}), \quad
  o = (o^{\mathrm{pub}}, o^{\mathrm{sec}})
,\]
there exists a predictor function
\[
  f_\textsf{predict}
  :
  S^{\mathrm{pub}} \times (I^{\mathrm{pub}})^* \to (O^{\mathrm{pub}})^*
,\]
where $\mathsf{pub\_inputs}(\vec{i})$ is the elementwise projection of an input sequence
to public components, and $\mathsf{pub\_outputs}(\mathcal{M}, s_0, \vec{i})$ is the corresponding projected observation trace.
Then, for every initial state $s_0 \in S$ and input sequence $\vec{i} \in I^*$,
\[
  \mathsf{pub\_outputs}(\mathcal{M},s_0,\vec{i})
  =
  f_\textsf{predict}\bigl(s_0^{\mathrm{pub}},\mathsf{pub\_inputs}(\vec{i})\bigr).
\]
\end{definition}

This definition originally comes from work on retrofitting the Bedrock2 compiler to prove preservation of cryptographic constant time across compiler passes~\cite{conoly2025}.
Intuitively, the observable behavior is fully determined by the public projection of the input and the initial state.
The definition above is a one-copy specification style: each run is characterized independently without needing to relate two separate sequences.
This definition is useful for certain proof techniques that can existentially construct a nice predictor function ($f_\textsf{predict}$).
The hardware development in this thesis uses the same one-copy perspective but restricted to state-machine traces with explicit input/output-erasure functions that capture the notion of partitioning state into secret and public components.

\subsection{Hardware constant time} \label{sec:hct}
Hardware constant time is motivated by hardware-software contracts~\cite{guarnieri2021}: the security guarantee is stated relative to an explicit attacker observation model. In that line of work, the contract ties together what information the attacker can observe (leakage events) and what information the system must hide as a function of secret inputs. In this thesis, we adopt the same pattern: we first make the attacker's observational capabilities explicit via erasure contracts (\Cref{def:erasure-spec}) and then define hardware constant time as an invariance of the erased observation when secret (erased) inputs vary.

\begin{definition}[Erasure contract] \label{def:erasure-spec}
For a state machine $\mathcal{M}=(S,s_0,I,O,\delta)$, an \emph{erasure contract} is a pair $\mathcal{E} = (\mathsf{erase_{in}}, \mathsf{erase_{out}})$ where $\mathsf{erase_{in}}$ is a function that erases secret components of inputs $\iota \in I$, and $\mathsf{erase_{out}}$ is a function that erases secret components of outputs $o \in O$ from the observational trace.
\end{definition}

\begin{definition}[Erased observation trace] \label{def:erased-run-outputs}
Given $\mathcal{M}$, $\mathcal{E}$, and an input sequence $\vec\iota \in I^*$, we define
$\mathcal{T}_{\mathcal{E}}\bigl( \mathcal{M}, \vec\iota \bigr)$ as the \emph{erased observation trace}, which is the observation trace $\mathcal{T}\bigl( \mathcal{M}, \vec\iota \bigr)$ with private output components deleted.
\end{definition}

\begin{definition}[Erased input sequence] \label{def:erased-input-sequence}
Given $\mathcal{M}$, $\mathcal{E}$, and an input sequence $\vec\iota \in I^*$, we define
$\vec\iota_\mathcal{E}$ as the \emph{erased input sequence}, which is the input sequence with private input components deleted.
\end{definition}

\begin{definition}[Hardware constant time] \label{def:hct-single}
A state machine $\mathcal{M}$ is \emph{hardware-constant-time} (HCT) with respect to an erasure contract $\mathcal{E}$ when
\[
  \forall \vec\iota \in I^*.\;\;
  \mathcal{T}_\mathcal{E}(\mathcal{M}, \vec\iota)
  =
  \mathcal{T}_\mathcal{E}(\mathcal{M}, \vec\iota_\mathcal{E}).
\]
\end{definition}

An erasure contract can be interpreted as a threat model, a description of the observational capabilities of an attacker.
It states which wires of a module's interface are observable to an attacker and which parts are treated as secret.
Concretely, $\mathsf{erase_{in}}$ controls observability on input wires: it preserves attacker-visible components and erases the rest.
Similarly, $\mathsf{erase_{out}}$ controls observability on output wires: it keeps attacker-visible outputs and removes or abstracts outputs that are outside the attacker's view.
Under this interpretation, HCT says that varying secret (erased) inputs cannot change the attacker-visible output trace.

\subsection{Writing proofs} \label{sec:timing-proof-lockstep}
The hardware-constant-time property (\Cref{def:hct-single}) reasons about entire runs of state machines rather than isolated steps.
Entire-run reasoning affords flexibility in potential proof techniques that may not reason about each step in isolation, but in the future it could allow simulation-based proofs that rely on skipping steps or taking multiple steps.
We will restrict ourselves to reason only about \emph{lockstep} simulations, which disallow skipping or multiple steps.
\Cref{fig:sim-machines} demonstrates the two different proof techniques introduced in this section: first, a machine compared against itself under erased inputs (self-simulation); and second, an implementation machine compared against an abstract specification (cross-simulation).

\paragraph{Setup}
Fix state machines $\mathcal{M}_1$ and $\mathcal{M}_2$ and erasure contract $\mathcal{E}$ as the following
\begin{gather*}
  \mathcal{M}_1 = (S_1, s_{0,1}, I, O, \delta_1), \qquad
  \mathcal{M}_2 = (S_2, s_{0,2}, I, O, \delta_2),\\
  \mathcal{E} = (\mathsf{erase_{in}},\mathsf{erase_{out}}).
\end{gather*}
Fix relation $R \subseteq S_1 \times S_2$. Let $a_1$ and $a_2$ be actions over $S_1$ and $S_2$, respectively, sharing output type $O$.
For related starting states $(s_1,s_2)\in R$, suppose each action steps to a successor state and output:
\[
  a_1(s_1)=(s_1',o_1),\qquad a_2(s_2)=(s_2',o_2).
\]

\begin{definition}[Erased-result relation] \label{def:erased-relation}
Given a relation $R \subseteq S_1 \times S_2$ and erasure contract $\mathcal{E}$, we say $R_\mathcal{E} \subseteq S_1 \times O \times S_2 \times O$ is the \emph{erased-result relation} when
\[
  R_\mathcal{E}\bigl(s_1,o_1,s_2,o_2\bigr)
  \;\triangleq\;
  \mathsf{erase_{out}}(o_1)=\mathsf{erase_{out}}(o_2) \; \land \; R(s_1,s_2).
\]
\end{definition}

\begin{definition}[Action simulation] \label{def:action-refinement}
We say $a_1 \sim a_2$ (read \emph{simulates}) under \((R,\mathcal{E})\) at \((s_1,s_2)\) when
\[
  R(s_1,s_2) \Rightarrow R_\mathcal{E}\bigl(s_1',o_1,s_2',o_2\bigr).
\]
This condition is depicted as a commutative diagram in~\Cref{fig:sim-action-square}.
\end{definition}

\begin{figure}[H]
\centering
\begin{tikzcd}[column sep=8em,row sep=huge]
  s_1 \arrow[r,"a_1"] \arrow[d,dashrightarrow,"R"'] & (s_1',o_1) \arrow[d,dashrightarrow,"R_\mathcal{E}"']
    \\
  s_2 \arrow[r,"a_2"'] & (s_2',o_2)
\end{tikzcd}
\caption[Action refinement]{Action refinement for \(a_1\) and \(a_2\) as a commutative diagram.}
\label{fig:sim-action-square}
\end{figure}

Next, machine simulation instantiates~\Cref{def:action-refinement} on every method invocation $\iota \in I$ where the abstract machine observes~\(\mathsf{erase_{in}}(\iota)\) wherever the implementation machine observes~\(\iota\).

\begin{definition}[Machine simulation] \label{def:machine-refinement}
$\mathcal{M}_1 \sim \mathcal{M}_2$ (read \emph{simulates}) \emph{under} $(R, \mathcal{E})$ when, for all related states $(s_1,s_2)\in R$, the following input-indexed action simulation holds:
\[
  \forall \iota \in I.\;\;
  \bigl(\lambda s.\; \delta_1(s,\iota)\bigr)
  \sim
  \bigl(\lambda s.\; \delta_2(s,\mathsf{erase_{in}}(\iota))\bigr)
  \text{ under } (R,\mathcal{E})
  \text{ at } (s_1,s_2)
.\]
\end{definition}

\paragraph{Self-simulation and cross-simulation}
The self-simulation and cross-simulation instances of this definition are shown in~\Cref{fig:sim-machines}. In self-simulation, the same machine appears on both sides: one side consumes the original input \(\iota\), while the other consumes the erased input \(\mathsf{erase_{in}}(\iota)\). Self-simulation directly matches the shape of hardware constant time, where we compare a machine against itself under secret variation. In cross-simulation, the left side is a concrete implementation, and the right side is a simpler abstract specification that already enforces the intended security behavior; the proof then factors through simulation to transfer that behavior back to the implementation.

\begin{figure}[H]
\centering
\begin{minipage}[t]{0.49\textwidth}
\centering
\textbf{(a) Self-simulation}\\
\smallskip
\begin{tikzcd}[column sep=8em,row sep=huge]
  s \arrow[r,"{\delta_{\mathcal{M}}(\iota)}"] \arrow[d,dashrightarrow,"R"'] & (s', o) \arrow[d,dashrightarrow,"R_\mathcal{E}"] \\
  \tilde{s} \arrow[r,"{\delta_{\mathcal{M}}(\mathsf{erase_{in}}(\iota))}"'] & (\tilde{s}', \tilde{o})
\end{tikzcd}
\end{minipage}\hfill
\begin{minipage}[t]{0.49\textwidth}
\centering
\textbf{(b) Cross-simulation}\\
\smallskip
\begin{tikzcd}[column sep=8em,row sep=huge]
  s_1 \arrow[r,"{\delta_{\mathcal{M}_{1}}(\iota)}"] \arrow[d,dashrightarrow,"R"'] & (s_1', o_1) \arrow[d,dashrightarrow,"R_\mathcal{E}"] \\
  s_2 \arrow[r,"{\delta_{\mathcal{M}_{2}}(\mathsf{erase_{in}}(\iota))}"'] & (s_2', o_2)
\end{tikzcd}
\end{minipage}
\caption[Self- and cross-simulation]{Lockstep simulation diagrams. Panel~\textbf{(a)} reuses~\(\delta_{\mathcal{M}}\) whereas panel~\textbf{(b)} contrasts~\(\mathcal{M}_{1}\) with~\(\mathcal{M}_{2}\).}

Next, we turn the stepwise lockstep obligations from~\Cref{fig:sim-action-square,fig:sim-machines} into trace consequences. We first define output-erased trace equivalence and show that machine simulation lifts to equality of erased traces. We then use these lemmas to derive the two proof patterns for hardware constant time: direct self-simulation and cross-simulation via a trace-equivalent specification.

\begin{definition}[Erased-trace equivalence] \label{def:outputs-erased-bridge}
Call~\(\mathcal M_2\) \emph{erased-trace equivalent} to~\(\mathcal M_1\) under~\(\mathcal E\) when, for every~\(\vec\iota\in I^*\), the erased-input histories~\(\vec\iota_{\mathcal E}\) induce the same erased observation trace:
\[
  \mathcal{T}^{\mathcal E}(\mathcal M_1,\vec\iota_{\mathcal E})
  =
  \mathcal{T}^{\mathcal E}(\mathcal M_2,\vec\iota_{\mathcal E}).
\]
\end{definition}

\begin{lemma}[Simulation implies erased-trace equivalence]
\label{lem:sim-lift-erased-traces}
If \(\mathcal{M}_1\) simulates \(\mathcal{M}_2\) under~\((R, \mathcal E)\) in the sense of~\Cref{def:machine-refinement} and \(R(s_{0,1},s_{0,2})\), then for every input history~\(\vec\iota\in I^*\),
\[
  \mathcal{T}_{\mathcal E}(\mathcal M_1,\vec\iota)
  =
  \mathcal{T}_{\mathcal E}(\mathcal M_2,\vec\iota_{\mathcal E}).
\]
\end{lemma}

\begin{proof}[Proof]
Induct on~\(\vec\iota\); the base case on the empty list is immediate. The inductive case first applies~\Cref{def:machine-refinement} to the head of the trace before appealing to the induction hypothesis on the tails.
\end{proof}

\begin{lemma}[Self-simulation implies hardware constant time]
\label{lem:self-sim-hct}
If~\(\mathcal M\) simulates itself under~\((R,\mathcal E)\) and \(R(s_0,s_0)\), then~\(\mathcal M\) satisfies hardware constant time (\Cref{def:hct-single}) with respect to~\(\mathcal E\).
\end{lemma}

\begin{proof}[Proof]
Instantiate~\Cref{lem:sim-lift-erased-traces} with the specification and implementation both taken to be~\(\mathcal M\); the resulting equation is~\Cref{def:hct-single}.
\end{proof}

\begin{lemma}[Cross-simulation with trace equivalence implies hardware constant time]
\label{lem:cross-sim-hct}
Suppose \(\mathcal{M}_{\mathsf I}\) simulates \(\mathcal{M}_{\mathsf S}\) under~\((R,\mathcal E)\), that \(R(s_{0,\mathsf I},s_{0,\mathsf S})\), and that~\(\mathcal{M}_{\mathsf S}\) is erased-trace equivalent to~\(\mathcal{M}_{\mathsf I}\), as stipulated in~\Cref{def:outputs-erased-bridge}.
Then \(\mathcal{M}_{\mathsf I}\) satisfies hardware constant time (\Cref{def:hct-single}) with respect to~\(\mathcal E\).
\end{lemma}

\begin{proof}[Proof]
Instantiate~\Cref{lem:sim-lift-erased-traces} with the implementation taken to be $\mathcal{M}_{\mathsf I}$ and the specification taken to be $\mathcal{M}_{\mathsf S}$. Proof obligations to show $\mathcal{M}_{\mathsf I}$ simulates $\mathcal{M}_{\mathsf S}$ under $(R, \mathcal{E})$ and that $R(s_{0,\mathsf I}, s_{0,\mathsf S})$ are provided as assumptions of the lemma.
\end{proof}

\chapter{Verifying correctness for circuits} \label{chap:Golemma}

This chapter presents a framework for verifying functional correctness on synthesizable hardware circuits. It first introduces the deeply embedded Golemma language, which has similarities to Bluespec SystemVerilog (\Cref{sec:Golemma-intro}).
Then, it covers language extensions that enable the synthesis of Gallina types and for loops into Golemma (\Cref{sec:Golemma-synth}).
Finally, it discusses how functional specifications are written and develops the proof strategy used to show modular refinement (\Cref{sec:Golemma-spec}).

\section{Encoding circuits with Golemma} \label{sec:Golemma-intro}
This section introduces the key constructs of Golemma through an example circuit: a module that computes the greatest common divisor (GCD) of two 16-bit numbers using the Euclidean algorithm.
The interface and module definition are shown in~\Cref{fig:Golemma-gcd-intf,fig:Golemma-gcd}, respectively.

\subsection{Interfaces}
An \emph{interface} in Golemma is a typed list of method signatures that a module presents to its environment.
Interfaces serve as contracts in two complementary senses. First, interfaces defines the compositional boundary between a module and its environment, playing the same role as the port list of a Verilog module. Second, it constrains what can be verified, since any specification must be stated in terms of the same method signatures that the implementation exposes.

The interface for the GCD module (\Cref{fig:Golemma-gcd-intf}) declares four methods.
Two are \emph{action methods}: $\mathit{load}$ and $\mathit{step}$, which can write registers and may abort if their guards are not met; their return type is $\mathit{Action}$.
Two are \emph{value methods}: $\mathit{result}$ and $\mathit{finished}$, which are purely observational  and cannot modify registers.

\begin{figure}[H]
\begin{minted}{coq}
interface GCD;
  method  Action      load  <Bit#(16) a, Bit#(16) b>;
  method  Action      step  <>;
  method  Bit#(16)    result <>;
  method  Bit#(1)     finished  <>;
endinterface.
\end{minted}
\caption[GCD interface]{The GCD interface with two action methods and two value methods.}
\label{fig:Golemma-gcd-intf}
\end{figure}

\subsection{Modules}
A \emph{module} instantiates an interface.
Modules contain \emph{submodules} with which they interact through method invocations defined in interfaces.
Registers are implemented as submodules: a register submodule holds state across clock cycles and exposes methods for reading and updating that state.
\Cref{fig:Golemma-gcd} shows the implementation of the GCD module.
The header declares that the module implements the $\mathit{GCD}$ interface.
The two register declarations \mintinline{coq}{Reg#(Bit#(16)) x <- mkReg()} and \mintinline{coq}{Reg#(Bit#(16)) y <- mkReg()} create the persistent state.
Submodule interactions, including register reads and writes, use backtick notation, as in \mintinline{coq}{`x`read()} and \mintinline{coq}{`x`write(v)}.
Unlike most register-transfer-level languages, Golemma does not have the restriction that every read within a method body sees the value the register held at the start of the method call; it sees the last write that occurred.

\begin{figure}[t]
\begin{minted}{coq}
module mkGCD#(GCD);
  Reg#(Bit#(16))  x  <- mkReg();
  Reg#(Bit#(16))  y  <- mkReg();

  method Action load <Bit#(16) a, Bit#(16) b>;
    begin
      `x`write(a);
      `y`write(b);
    end.
  endmethod.

  method Action step <>;
    begin
      let xv = `x`read();
      let yv = `y`read();
      if (yv == 16'd0)  abort;
      if (xv >= yv)     `x`write(xv - yv);
      else              `y`write(yv - xv);
    end.
  endmethod.

  method Bit#(16) result <>;
    begin return `x`read(); end.
  endmethod.

  method Bit#(1) finished <>;
    begin return (`y`read() == 16'd0); end.
  endmethod.
endmodule.
\end{minted}
\caption[GCD module]{A GCD module written in Golemma.}
\label{fig:Golemma-gcd}
\end{figure}

\subsection{Methods}
Every module method is declared with a return type, a name, and an optional parameter list enclosed in angle brackets.
Methods whose return type are $\mathit{Action}$ or \mintinline{coq}{ActionValue\#(t)} may call submodule action methods; they can also abort, leaving state unchanged.
Methods with any other return types are \emph{value methods}: they may only read state and return pure values, and they never abort.
This distinction is enforced syntactically: only action methods may contain write or abort statements, so the separation between observation and mutation is visible directly in the interface signature.

\subsection{Expressions}
Expressions are pure terms of a given type.
The primitive types in Golemma are bitvectors ($\mathit{Bit\#(n)}$ for a $n$-bit word), fixed-size vectors of types ($\mathit{Vector\#(n,t)}$ for a fixed-size array of $t$ with $n$ items), structs, and enums.
An expression can be a constant literal such as \mintinline{coq}{16'd0}, a local variable bound by a prior $\mathit{let}$ statement, a field access \mintinline{coq}{e`f}, a primitive operator such as addition or comparison, or a submodule call \mintinline{coq}{`sub`met(args)}.
Value method calls read current register state inside a submodule and return a result without modifying it; they appear as expressions because their effect are purely observational.

\subsection{Statements}
Statements sequence effects inside a method body.
The \mintinline{coq}{let x = e} form binds a fresh variable $\mathit{x}$ to the value of expression $\mathit{e}$ in the rest of the block.
A typed form \mintinline{coq}{T\#($t$) x = e} binds $\mathit{x}$ with an explicit type annotation.
Register state is updated via submodule action method calls: \mintinline{coq}{`reg`write(e)} calls the $\mathit{write}$ action on register $\mathit{reg}$ with argument $\mathit{e}$.
The result of calling an action-method can be captured with \mintinline{coq}{let x <- `mod`met(args)}.
Conditional branching uses \mintinline{coq}{if (c) { ... } else { ... }}, where $\mathit{c}$ is an expression of type \mintinline{coq}{Bit#(1)}; the $\mathit{else}$ branch is optional and defaults to a no-op.
A statement may be terminated early by $\mathit{abort}$, which causes the entire method to fail and appear as if it never exited.
No special synthesis is needed to implement $\mathit{abort}$, but it induces a precondition on the method.
The method only will be called in states where execution cannot reach the $\mathit{abort}$.
During verification, callers must prove that this precondition holds.
Statements are sequenced by separating them with semicolons inside a $\mathit{begin}$ ... $\mathit{end}$ block.
A block may end with a \mintinline{coq}{return e} expression that specifies the method's output value; methods of type $\mathit{Action}$ whose return type are $\mathit{unit}$ may omit $\mathit{return}$ and close with $\mathit{end}$ directly.

\section{Extending Golemma with type classes} \label{sec:Golemma-synth}

This section discusses a language extension that enables the reification of finite Gallina types and terms into Golemma (\Cref{sec:Golemma-synthesizable}).
It also discusses how this extension enables other language features such as structs (\Cref{sec:Golemma-struct}), enums (\Cref{sec:Golemma-enum}), bitflags (\Cref{sec:Golemma-bitflag}), and for loops (\Cref{sec:Golemma-for}).

\subsection{Synthesizable types} \label{sec:Golemma-synthesizable}

Golemma programs cannot directly use standard Rocq types such as plain records and inductive types; instead, they must use the types provided by Golemma's type system.
Writing functional specifications and invariants that reason directly over Golemma types is possible but cumbersome.
Proofs would need to reason about raw bitvectors rather than natural Gallina types, which are the easiest to work with and the most performant with the type checker in proofs.
A better approach is to allow the developer to use familiar Gallina types directly in Golemma circuits, so that specifications and proofs can be stated at the Gallina level and then connected to the hardware representation transparently.

The Synthesizable type class formalizes this connection.
Instances of the $\mathit{Synthesizable}$ type-class witness that a Gallina type $\alpha$ has a corresponding Golemma hardware type $\tau$ and provide the two functions that convert between them:
\begin{minted}[mathescape=true,escapeinside=||]{coq}
Class Synthesizable (|$\alpha$| : Type) := {
  (* [type] is the type of embedded Golemma types *)
  |$\tau$|           : type;
  pack        : |$\alpha$| -> type_denote |$\tau$|;
  unpack      : type_denote |$\tau$| -> |$\alpha$|;
  pack_unpack : forall x, unpack (pack x) = x;
}.
\end{minted}
The field $\tau$ is the Golemma type representing $\alpha$ in hardware.
The function $\mathit{pack}$ is an injection from $\alpha$ into the hardware representation: $\mathit{unpack}$ is its left inverse, recovering the original Gallina value from the encoding, and $\mathit{pack\_unpack}$ proves that $\mathit{pack}$ is indeed an injection.
Every native Golemma type is trivially synthesizable with \mintinline{coq}{pack = unpack = id}.
Beyond Golemma types, several finite Gallina types have built-in instances: bitvectors \mintinline{coq}{bv n}, bounded naturals \mintinline{coq}{fin n}, Booleans $\mathit{bool}$, and $\mathit{unit}$.
Three additional type classes extend $\mathit{Synthesizable}$ to structured Gallina types: Struct, Enum, and Bitflags.
Each provides a Synthesizable instance automatically during type-class resolution.

\subsection{Structures} \label{sec:Golemma-struct}
The Struct type class handles Gallina records whose fields are all synthesizable.
An instance maps the record to a Golemma deeply embedded struct type, with each field packed and unpacked field-by-field.
\Cref{fig:synth-struct} shows a simple example.
The tactic $\mathit{derive\_struct}$ inspects the definition of the record using Ltac2 metaprogramming (\Cref{par:ltac2}), extracts the field names and types, constructs the packing and unpacking mechanism, and discharges all proof obligations automatically.

\begin{figure}[H]
\begin{minted}{coq}
Record TwoBits := {
  bit0 : bool;
  bit1 : bool;
}.

Instance two_bits_struct : Struct TwoBits. derive_struct. Defined.
\end{minted}
\caption[Struct instance]{Deriving a $\mathit{Struct}$ instance for a simple record.}
\label{fig:synth-struct}
\end{figure}

\subsection{Enumerables} \label{sec:Golemma-enum}
The Enum type class handles Gallina inductive types with only nullary constructors, that is, simple variant types with no associated data.
An instance maps each constructor to a distinct element of a Golemma $\mathit{enum\_t}$, with the representation width chosen as $\lceil \log_2 n \rceil$ bits for a type with $n$ constructors.
\Cref{fig:synth-enum} shows a three-constructor color type.
The tactic $\mathit{derive\_enum}$ automatically computes the constructor names, bitvector representations, and all proof obligations.

\begin{figure}[H]
\begin{minted}{coq}
Variant Color : Set := Red | Green | Blue.

\subsection{Metaprogramming} \label{par:ltac2}
In proof assistants with foundations in type theory, such as Rocq, tactics are scripts that used to concisely construct proof terms.
Ltac1 is Rocq's original and most commonly used tactic language.
Ltac2 is a newer, typed tactic language with better support for inspecting and constructing Rocq terms programmatically, making it a better fit for metaprogramming purposes.

Deriving instances for $\mathit{Struct}$, $\mathit{Enum}$, and $\mathit{Bitflags}$ requires inspecting inductive type definitions at elaboration time.
Rather than requiring these type-class instances to be derived by hand, the framework uses Ltac2 to perform this metaprogramming.
The derivation tactics ($\mathit{derive\_struct}$, $\mathit{derive\_enum}$, $\mathit{derive\_bitflags}$) each reflect a Rocq inductive into a Golemma type signature and discharge any resulting proof obligations using Ltac1.

When a Golemma expression contains an identifier, the language front-end uses a sophisticated type-class-resolution mechanism to decide if it refers to a local method parameter, a local let binding, or a submodule.
To add support for referring to record projections or Gallina constructors, this system was extended to recognize identifiers related to synthesizable types.
This resolution is handled by a hint in the type-class resolution database that fires whenever the elaborator encounters a particular type-class goal that resolves identifiers.
The hint, written in Ltac2, recovers the identifier name as a Rocq string, searches first among local hypotheses and then the global environment for a binding with that name, and, if found, wraps the value in \mintinline[escapeinside=||]{coq}{Cst (pack |$\cdot$|)}: the syntax tree for a constant node in Golemma.
The result is that any Gallina constant or local variable of a synthesizable type can be dropped into a Golemma expression directly by name, without any explicit coercion syntax.

\subsection{For loops} \label{sec:Golemma-for}

Hardware circuits commonly need to operate over fixed-size arrays of registers, for example, to implement content-addressable memories (CAMs).
Golemma did not provide native syntax for for-loop iteration at the time this thesis was written.  The for constructor in the syntax tree takes a bound $m$ and a body function from \mintinline{coq}{Fin m} to a block of actions, executing the body once for each index in \(0, 1, \ldots, m-1\).

The key step is converting the index \mintinline{coq}{i : Fin m} into a bitvector of size $\lceil \log_2{m} \rceil$ so that the body can use it in Golemma expressions directly.
This conversion process happens automatically via the type-class-resolution procedure described in \cref{sec:Golemma-synthesizable,par:ltac2}.
A custom notation then exposes a familiar loop syntax as a Golemma statement:
\begin{minted}{coq}
for (let i = 0 to hi) begin
  (* body, i : bv hi *)
end
\end{minted}
The loop bound $\mathit{hi}$ must be a natural number fixed at elaboration time, so the loop always runs a statically known number of iterations, as required for synthesizability. During verification, for-loop semantics reduce to a loop rule that requires an explicit loop invariant.

\section{Specifying and proving correctness} \label{sec:Golemma-spec}

Proving that a Golemma module is correct requires a way to express what the module is supposed to do and a way to connect that description to the module's behavior.
This section describes both.
It first introduces how to write relational specifications for Golemma methods and rules that are then composed into a modular specification (\Cref{sec:Golemma-spec-framework}).
Finally, it discusses the proof strategy needed to show that the implementation refines the specification (\Cref{sec:Golemma-spec-refinement}).

\subsection{Writing specifications} \label{sec:Golemma-spec-framework}

A specification describes the set of valid behaviors a module may exhibit.
Specifications do not need to be executable; they are relational, and it is reasonable for them to use existential quantifiers, which are expressive and enable nondeterminism naturally.
The only constraint is that a specification must cover the same interface as the implementation it is meant to describe. The interface acts as the specification boundary, and both the concrete module and its specification provide an implementation of every method declared there, with matching argument and return types.
The rest of this subsection develops a specification for the greatest-common-divisor module from \Cref{fig:Golemma-gcd-intf}, using it to illustrate the key features of this specification style.

\paragraph{Abstract state.}
Every specification module has an associated \emph{abstract state}, which is an ordinary Gallina type chosen to model the module's behavior at the desired level of abstraction.
For the GCD module, the abstract state records both the original inputs and the current state of the Euclidean algorithm.
The fields $\mathit{a}$ and $\mathit{b}$ are the original operands loaded into the module, while $\mathit{a'}$ and $\mathit{b'}$ are the current operands after some number of $\mathit{step}$ calls.
The \emph{representation invariant} ($\mathit{wf}$) states that the current operands still have the same greatest common divisor as the original inputs.

\begin{minted}{coq}
Record SpecState := mkState {
  a  : nat;
  b  : nat;
  a' : nat;
  b' : nat;
  wf : gcd a b = gcd a' b';
}.
\end{minted}

The abstract state lives entirely in Gallina; it is only the implementation that requires the use of the hardware-description language.
This specification style is a benefit: proofs, invariants, and lemmas about the specification are stated in terms of natural-number arithmetic and the standard GCD function rather than in terms of bitvectors.

\paragraph{Predicate-transformer semantics.}
Each method specification is written in a continuation-passing style.
Rather than producing an output value directly, a specification takes a postcondition $P$ and is responsible for establishing that $P$ holds of its outputs.
This continuation-passing convention gives a uniform way to state both what the method requires of its inputs (preconditions) and what it guarantees about its outputs (postconditions).
Golemma provides two specification types.
Value-method specifications have the following type:
\[
  (\mathit{args} \times \mathit{State}) \to (\mathit{ret} \to \mathsf{Prop}) \to \mathsf{Prop}
\]
Action-method specifications have the following type, where the postcondition additionally receives the next state:
\[
  (\mathit{args} \times \mathit{State}) \to ((\mathit{ret} \times \mathit{State}) \to \mathsf{Prop}) \to \mathsf{Prop}
\]
In both cases the outermost proposition is the body of the specification.

\paragraph{Value methods.}
The $\mathit{result}$ method returns the correct result from the GCD computation and $\mathit{finished}$ returns whether the current second operand $\mathit{b'}$ is zero.
Each specification simply calls $P$ on the appropriate return value:
\begin{minted}{coq}
Definition result : MRVSpec State (mkMetSig <> Bit#(16)) :=
  fun '(_, st) P =>
    st.(b') = 0 /\ P (nat_to_bv (gcd st.(a) st.(b))).

Definition finished : MRVSpec State (mkMetSig <> Bit#(1)) :=
  fun '(_, st) P =>
    P (if (st.(b') == 0) then 1%bv else 0%bv).
\end{minted}
The postcondition is stated by choosing what term to apply $P$ to.
Different terms produce stronger or weaker guarantees; the specification's job is to apply $P$ to exactly the value the method is guaranteed to return.

\paragraph{Action methods and preconditions.}
Action-method specifications additionally describe how the abstract state evolves.
The $\mathit{load}$ method sets the two operands to the supplied arguments:
\begin{listing}[H] % [H] forces it to stay "Here"
\begin{minted}[escapeinside=||]{coq}
Definition load : MRASpec State (mkMetSig <Bit#(16) a, Bit#(16) b> Bit#(0)) :=
  fun '(args, _) P =>
    let a := bv_to_nat args[[@0]] in
    let b := bv_to_nat args[[@1]] in
      P (0%bv, mkState a b a b _).
\end{minted}
\end{listing}
The successor state stores the loaded operands twice: once as the fixed original inputs and once as the current operands.
The proof witness $\mathit{pf}$ establishes \mintinline{coq}{gcd a b = gcd a b}, so the invariant holds immediately after loading.

The $\mathit{step}$ method is more interesting because the implementation aborts when the current second operand is zero.
A \emph{precondition} on the method is expressed as a conjunct placed \emph{outside} of $P$, before any existential quantification.
\begin{minted}{coq}
Definition step : MRASpec State (mkMetSig <> Bit#(0)) :=
  fun '(_, st) P =>
    st.(b') <> 0 /\
    let next :=
      if decide (st.(a') >= st.(b')) then
        mkState st.(a) st.(b) (st.(a') - st.(b')) st.(b') _
      else
        mkState st.(a) st.(b) st.(a') (st.(b') - st.(a')) _ in
      P (0%bv, next).
\end{minted}
Since \mintinline{coq}{st.(b') <> 0} sits outside $P$, it is a delegated obligation that callers of this submodule must discharge.
The refinement proof must show that the implementation never invokes $\mathit{step}$ in a state where the abort path can be reached.
Postconditions, by contrast, are captured by constructing the next abstract state passed to $\mathit{P}$; here that state preserves the original $\mathit{a}$ and $\mathit{b}$ fields while updating only $\mathit{a'}$ and $\mathit{b'}$.

\paragraph{Representation invariants.}
Not every tuple of natural numbers is a valid intermediate state for a GCD computation.
The $\mathit{SpecState}$ record shown earlier encodes this constraint through the $\mathit{wf}$ field: a Rocq proof that the current pair \mintinline{coq}{(a', b')} has the same GCD as the original pair \mintinline{coq}{(a, b)}.
This proof field is a \emph{representation invariant} that every abstract state must carry.

Since $\mathit{wf}$ is a field of the record, Rocq's type system ensures it is present in every $\mathit{SpecState}$ value. The type system disallows constructing an abstract state that violates it.
Action-method specifications that modify the state have a proof obligation to discharge the invariant.
For example, the $\mathit{step}$ specification must supply a proof that the updated current pair still has GCD \mintinline{coq}{gcd st.(a) st.(b)} whenever it constructs the next $\mathit{SpecState}$.

\paragraph{Connection to abstract data types.}
This specification style is directly analogous to the theory of \emph{abstract data types} (ADTs).
In an ADT, an abstraction function (AF) maps concrete states to abstract values, and each operation is specified by its effect on that abstract model.
The same structure appears here: the $\mathit{SpecState}$ record is the abstract model, and each method specification describes the contract that any correct implementation must satisfy.
The predicate-transformer style mirrors Hoare-logic-style method contracts.
A precondition is a proposition outside $P$ that constrains what the method requires of its inputs; a postcondition is a proposition inside $P$ that describes what the method guarantees about its outputs.
Similar to ADT specifications, these contracts are \emph{relational}. Specifications are allowed to use arbitrary Gallina propositions, including existential quantification, so they can express nondeterminism cleanly.
An implementation that can nondeterministically return any element of a set satisfying some predicate is expressed by quantifying existentially over the return value. The specification makes no commitment about which element is chosen.
This feature is particularly useful when specifying the issue logic of an out-of-order processor scheduler, where the choice of which ready instruction to issue is left unspecified.
The abstraction function appears explicitly as the simulation relation between the concrete register state of the implementation and the abstract state, when establishing refinement.

\subsection{Writing proofs} \label{sec:Golemma-spec-refinement}

A functional-correctness proof in Golemma is a proof of \emph{module refinement}: every behavior the implementation can exhibit is permitted by the specification.
The proof is organized around a simulation relation $R$ that pairs implementation states with their abstract counterparts, and proceeds method by method, showing that each implementation method satisfies the postcondition demanded by its specification whenever $R$ holds of the current state.

\paragraph{Refinement.}
A simulation relation $R : I \to S \to \text{Prop}$ relates each implementation state $i \in I$ to an abstract specification state $s \in S$.
Throughout the definitions below, $\varphi$ denotes a predicate transformer for the implementation and $\psi$ denotes the corresponding predicate transformer for the specification, where each is a predicate transformer of the appropriate type from \Cref{sec:Golemma-spec-framework}.
Given such a relation, refinement comes in three variants: value-method refinement, action-method refinement, and for rule refinement.

\begin{definition}[Value-method refinement] \label{def:refines-value}
Let $\varphi$ and $\psi$ be value-method-predicate transformers over states $I$ and $S$, respectively.
$\varphi$ \emph{refines} $\psi$ under $R$ when, for every related pair $(i, s) \in R$, every argument $\alpha$, and every postcondition $P$,
\[
  \psi(\alpha,\, s,\, P) \;\Rightarrow\; \varphi(\alpha,\, i,\, P).
\]
\end{definition}

Value-method refinement is straightforward: if the specification establishes postcondition $P$, so must the implementation.
Action methods will update state, so their refinement condition must also account for how the two states evolve:

\begin{definition}[Action-method refinement] \label{def:refines-action}
Let $\varphi$ and $\psi$ be action-method-predicate transformers over states $I$ and $S$, respectively.
$\varphi$ \emph{refines} $\psi$ under $R : I \to S \to \text{Prop}$ when, for every $(i, s) \in R$, every argument $\alpha$, and every postcondition $P$,
\[
  \psi(\alpha,\, s,\, P)
  \;\Rightarrow\;
  \varphi(\alpha,\, i,\, \lambda\,(r,\,i').\;
    \exists\, s'.\; P(r,\, s') \land R(i',\, s')).
\]
\end{definition}

Rather than demanding a unique successor state, the existential allows the implementation to reach any state $i'$ for which there exists a specification successor $s'$ still in $R$ and satisfying $P$.
Rules carry no arguments and produce no return value, but they do update state.
Rule refinement therefore follows the same shape as action-method refinement, with the added flexibility of a possibly different post-relation $R'$ for relating successor states:

\begin{definition}[Rule refinement] \label{def:refines-rules}
Let $\varphi$ and $\psi$ be rule-predicate transformers over states $I$ and $S$, respectively.
$\varphi$ \emph{refines} $\psi$ under $(R, R' : I \to S \to \text{Prop})$ when, for every $(i, s) \in R$ and every postcondition $P$,
\[
  \psi(s,\, P)
  \;\Rightarrow\;
  \varphi(i,\, \lambda\, i'.\;
    \exists\, s'.\; P(s') \land R'(i',\, s')).
\]
\end{definition}

The use of $R'$ as the post-relation separates the invariant needed to fire a rule from the invariant that holds once it has fired, which is useful when rules have side-conditions that are established by other rules in the same module.

Module refinement packages all of the method-level and rule-level obligations together with a proof that the reset states are related:

\begin{definition}[Module refinement] \label{def:refines-module}
A module $\mathcal{M}_I$ \emph{refines} a module $\mathcal{M}_S$ when there exist simulation relations $R, R' : I \to S \to \text{Prop}$ and initial states $i_0 \in I$, $s_0 \in S$ such that:
\begin{enumerate}
  \item $R(i_0, s_0)$ (the reset states are related);
  \item every external method of $\mathcal{M}_I$ refines the corresponding method of $\mathcal{M}_S$ under $R$; and
  \item every rule of $\mathcal{M}_I$ refines the corresponding rule of $\mathcal{M}_S$ under $(R, R')$.
\end{enumerate}
\end{definition}

\paragraph{Example.}
We walk through a complete functional-correctness proof for the GCD module as a worked example.
The proof obligations reduce to a collection of per-method-refinement lemmas.
The most important design decision for the proof is the choice of simulation relation~$R$.
The relation asserts that each register value, converted from a bitvector to a natural number, agrees with the corresponding abstract field.
\Cref{fig:gcd-R} shows the Rocq definition.
\begin{figure}[H]
\begin{minted}{coq}
Definition R (ist : ImplState) (sst : SpecState) : Prop :=
  bv_to_nat ist.(reg_a) = sst.(a') /\ bv_to_nat ist.(reg_b) = sst.(b').
\end{minted}
\caption[GCD simulation relation]{Simulation relation for the GCD module.}
\label{fig:gcd-R}
\end{figure}
With $R$ fixed, each method proof follows the same pattern: use the equalities in $R$ to rewrite the implementation's register values as their abstract counterparts, then show the resulting expression matches the postcondition $P$ of the specification.
Action-method proofs require exhibiting a successor abstract state and showing that $R$ is restored after the step.
The lemmas below demonstrate an example proof for each method of the GCD module.

\begin{lemma} \label{lem:gcd-zero}
For all $a$, $b$, $x$, if $\gcd(a, b) = \gcd(x, 0)$ then $x = \gcd(a, b)$.
\end{lemma}

\begin{lemma} \label{lem:gcd-subtract}
For all $a$, $b$, if $b \geq a$ then $\gcd(a, b) = \gcd(b, b - a)$.
\end{lemma}

\begin{lemma} \label{lem:gcd-swap}
For all $a$, $b$, $\gcd(a, b) = \gcd(b, a)$.
\end{lemma}

\begin{lemma}\label{lem:gcd-finished-refines}
$\mathit{finished}$ refines its method specification under $R$.
\end{lemma}

\begin{proof}[Proof]
Assume $(i, s) \in R$.
The implementation checks whether $i.b$ is zero; the specification checks whether the abstract field $s.b'$ is zero.
Since $R$ equates the two (modulo type conversion), both checks yield the same Boolean, so the implementation returns exactly the value the specification prescribes.
\end{proof}

\begin{lemma} \label{lem:gcd-result-refines}
 $\mathit{result}$ refines its specification under $R$, provided $s.b' = 0$.
Moreover, the value returned by $\mathit{result}$ equals $\gcd(s.a, s.b)$.
\end{lemma}

\begin{proof}[Proof]
Assume $(i, s) \in R$.
The specification gives the bitvector encoding of $s.a'$. Since $R$ equates $i.a$ with $s.a'$, the implementation's return value is equal to what the specification states.
The representation invariant gives $\gcd(s.a, s.b) = \gcd(s.a', 0)$, and \Cref{lem:gcd-zero} then concludes $i.a = s.a' = \gcd(s.a, s.b)$.
\end{proof}

\begin{lemma} \label{lem:gcd-load-refines}
$\mathit{load}$ refines its specification under $R$.
\end{lemma}

\begin{proof}[Proof]
Assume $(i, s) \in R$ and arguments $a_{\mathit{val}}$, $b_{\mathit{val}}$.
The specification produces a successor state in which all four fields are set to those argument values (the original inputs $a$ and $b$ and the current operands $a'$ and $b'$ are all initialized simultaneously).
The implementation writes the same two values into the corresponding registers.
Taking the successor abstract state to be the one the specification constructs, $R$ holds immediately from the register-write semantics, and the representation invariant $\gcd(a, b) = \gcd(a', b')$ reduces to $\gcd(a_{\mathit{val}}, b_{\mathit{val}}) = \gcd(a_{\mathit{val}}, b_{\mathit{val}})$, which is trivially true.
\end{proof}

\begin{lemma} \label{lem:gcd-step-refines}
$\mathit{step}$ refines its specification under $R$, provided $s.b' \neq 0$.
\end{lemma}

\begin{proof}[Proof]
Assume $(i, s) \in R$.
Since $R$ equates $s.b'$ with $i.b$, $i.b$ is also nonzero, so the implementation does not reach the $\mathit{abort}$ path.
Both the specification and the implementation perform the same conditional subtraction, and since the register values equal the abstract fields by $R$, the two conditionals take the same branch.
After the update, the new register values equal the new abstract fields, so $R$ is restored.

Now we must show that $\gcd(a, b) = \gcd(a'', b'')$ holds for the successor state $(a'', b'')$.
In the first branch, $s.a' \geq s.b'$, so $(a'', b'') = (s.a' - s.b',\; s.b')$.
The invariant $\gcd(a,b) = \gcd(s.a', s.b')$ and \Cref{lem:gcd-subtract} give $\gcd(s.a', s.b') = \gcd(s.a' - s.b', s.b')$, establishing the invariant.
In the second branch, $s.a' < s.b'$, so $(a'', b'') = (s.a',\; s.b' - s.a')$.
Applying \Cref{lem:gcd-swap} gives $\gcd(s.a', s.b') = \gcd(s.b', s.a')$, then \Cref{lem:gcd-subtract} gives $\gcd(s.b', s.a') = \gcd(s.b' - s.a', s.a')$, and a final application of \Cref{lem:gcd-swap} yields $\gcd(s.a', s.b' - s.a')$, establishing the invariant.
\end{proof}

\begin{theorem} \label{thm:gcd-correct}
The GCD implementation module refines the GCD specification module.
\end{theorem}

\begin{proof}[Proof]
Apply \Cref{def:refines-module}: the reset states are related by construction, and \Cref{lem:gcd-finished-refines,lem:gcd-result-refines,lem:gcd-load-refines,lem:gcd-step-refines} discharge the per-method obligations.
\end{proof}

\section{Timing security: Variable-latency multiplier} \label{sec:case-study-multiplier}

This section presents the timing-security verification of a variable-latency integer multiplier using Slate.
The multiplier computes the product of two 8-bit operands using a shift-and-add algorithm.
Since the number of cycles depends on the first operand and not the second, the second operand can be treated as a secret without affecting the timing behavior.
The goal is to establish hardware constant time with respect to an erasure contract that hides the second operand and the final product.

\subsection{Erasure contract}
The secrets are the second operand $\mathit{b}$ of $\mathit{enq}$ and the product $\mathit{c}$ of $\mathit{deq}$.

\subsection{Implementation}

The multiplier holds five registers: $\mathit{src1}$ and $\mathit{src2}$ (the two 8-bit operands), $\mathit{dst}$ (the 16-bit accumulator), $\mathit{count}$ (a 3-bit iteration counter), and $\mathit{phase}$.
\Cref{fig:mult-state} shows the state record.

\begin{figure}[ht]
\begin{minted}{coq}
Variant Phase := Empty | Busy | Full.

Record State := {
  src1  : Bit#(8);
  src2  : Bit#(8);
  dst   : Bit#(16);
  count : Bit#(3);
  phase : Phase;
}.
\end{minted}
\caption[Multiplier state type]{State type and phases of the multiplier.}
\label{fig:mult-state}
\end{figure}

Each method call to $\mathit{step}$ invokes the $\mathit{shift\_and\_add}$ subroutine defined in \Cref{fig:mult-shift-add}.
If $\mathit{src1}$ is already zero then $\mathit{shift\_and\_add}$ is a no-op. If the least significant bit of $\mathit{src1}$ is one, the subroutine adds $\mathit{src2}$ shifted left by $\mathit{count}$ to $\mathit{dst}$. If the least significant bit is instead zero, nothing is added to $\mathit{dst}$.
In both cases, $\mathit{src1}$ is right-shifted by one and $\mathit{count}$ is incremented.
Once $\mathit{src1}$ reaches zero, the outer $\mathit{step}$ transitions the phase to $\mathit{Full}$, making the result available for dequeue.

\begin{figure}[ht]
\begin{minted}{coq}
Definition shift_and_add :=
  begin
    let src1'  = `src1`read();
    let src2'  = `src2`read();
    let count' = `count`read();
    let dst'   = `dst`read();
    if (src1' == 0%bv)
      pass
    else begin
      let t =
        if (truncate1 src1' == 1)%bv then
          zero_extend src2'
        else
          (16`0x00)%bv;
      `dst`write((dst' + bv_shiftl t (zero_extend count'))%bv);
      `count`write((count' + 1)%bv);
      `src1`write(bv_shiftr src1' 1)
    end
  end.
\end{minted}
\caption[Shift-and-add subroutine]{The $\mathit{shift\_and\_add}$ subroutine. On each call it processes one bit of $\mathit{src1}$, conditionally adding a shifted copy of $\mathit{src2}$ to the accumulator.}
\label{fig:mult-shift-add}
\end{figure}

\subsection{Verification}

The central observation is that the timing of the multiplier depends entirely on $\mathit{src1}$: the number of steps before the phase transitions to $\mathit{Full}$ is exactly the position of the most significant set bit of $\mathit{src1}$.
The value of $\mathit{src2}$ affects only the accumulator $\mathit{dst}$ and therefore does not influence any externally observable signal.
This observation is captured by the simulation relation in \Cref{fig:mult-rel}, which equates the $\mathit{phase}$, $\mathit{count}$, and $\mathit{src1}$ registers of two states but makes no requirement on $\mathit{src2}$ or $\mathit{dst}$.

\begin{figure}[ht]
\begin{minted}{coq}
Definition R (st st' : State) : Prop :=
  st.(phase) = st'.(phase) /\
  st.(count) = st'.(count) /\
  st.(src1)  = st'.(src1).
\end{minted}
\caption[Multiplier simulation relation]{Simulation relation for the multiplier. Two states are related when they agree on the timing-relevant registers.}
\label{fig:mult-rel}
\end{figure}

The proof uses self-simulation (\Cref{lem:self-sim-hct}): the implementation is simulated against itself with the erased arguments and results.
It suffices to show that $R$ is preserved by each operation.

\begin{lemma} \label{lem:mult-enq}
$\mathit{enq}$ self-simulates under $(R, \mathcal{E})$.
\end{lemma}

\begin{proof}[Proof]
Assume $(s_1, s_2) \in R$.
The guard checks $s_1.\mathit{phase} = \mathit{Empty}$; since $R$ equates the phases, $s_2.\mathit{phase} = \mathit{Empty}$ as well, so both copies take the same branch.
When the guard succeeds, both copies set $\mathit{src1} := a$, $\mathit{dst} := 0$, $\mathit{count} := 0$, and $\mathit{phase} := \mathit{Busy}$.
The first copy additionally sets $\mathit{src2} := b$ while the erased copy sets $\mathit{src2} := 0$; since $R$ imposes no constraint on $\mathit{src2}$ or $\mathit{dst}$, the successor states remain related.
Both copies return void, and $\mathcal{E}$ erases void to void, so the observed outputs agree.
\end{proof}

\begin{lemma} \label{lem:mult-deq}
$\mathit{deq}$ self-simulates under $(R, \mathcal{E})$.
\end{lemma}

\begin{proof}[Proof]
Assume $(s_1, s_2) \in R$.
The guard checks $s_1.\mathit{phase} = \mathit{Full}$; since $R$ equates the phases, both copies take the same branch.
Both copies set $\mathit{phase} := \mathit{Empty}$ and return $\mathit{dst}$.
The products $s_1.\mathit{dst}$ and $s_2.\mathit{dst}$ may differ, but $\mathcal{E}$ erases the output by replacing the product with zero, so the observed outputs are equal.
In the successor states, $\mathit{phase}$, $\mathit{count}$, and $\mathit{src1}$ are unchanged, so $R$ is preserved.
\end{proof}

\begin{lemma} \label{lem:mult-rule}
$\mathit{step}$ self-simulates under $(R, \mathcal E)$.
\end{lemma}

\begin{proof}[Proof]
Assume $(s_1, s_2) \in R$.
The guard checks $s_1.\mathit{phase} = \mathit{Busy}$; since $R$ equates the phases, both copies take the same branch.
Inside the guard, both copies check whether $\mathit{src1}$ is zero; since $R$ equates $\mathit{src1}$, both take the same inner branch.
If $s_1.\mathit{src1} = 0$, both copies set $\mathit{phase} := \mathit{Full}$ and leave the remaining registers unchanged, so $R$ holds for the successor states.
Otherwise, both right-shift $\mathit{src1}$ by one and increment $\mathit{count}$ identically, preserving the $\mathit{src1}$ and $\mathit{count}$ components of $R$.
The accumulation into $\mathit{dst}$ may differ since $s_1.\mathit{src2}$ and $s_2.\mathit{src2}$ may differ, but $\mathit{dst}$ is not tracked by $R$, so $R$ is preserved in both branches.
\end{proof}

\begin{lemma} \label{lem:mult-status}
$\mathit{empty}$, $\mathit{busy}$, and $\mathit{full}$ each self-simulate under $(R, \mathcal E)$.
\end{lemma}

\begin{proof}[Proof]
Assume $(s_1, s_2) \in R$.
Each status query reads only $\mathit{phase}$, compares it against a fixed constant, and returns a Boolean.
Since $R$ equates the two phases, both copies return the same Boolean, leave the state unchanged, and produce equal outputs.
\end{proof}

\begin{theorem} \label{thm:mult-hct}
The multiplier satisfies hardware constant time with respect to the erasure contract $\mathcal E$.
\end{theorem}

\begin{proof}[Proof]
Apply \Cref{lem:self-sim-hct} using simultion relation $R$.
\Cref{lem:mult-enq,lem:mult-deq,lem:mult-rule,lem:mult-status} discharge the resulting proof obligations.
The reset state satisfies $R$ trivially since both copies of the reset record are identical.
\end{proof}

\section{Functional correctness: Ring buffer} \label{sec:case-study-ring-buffer}

This section presents the proof of correctness for a ring buffer,
The ring buffer holds entries of parameterized type $\varepsilon$ and supports a capacity of $2^N$ slots.
Beyond the standard enqueue and dequeue operations, the ring buffer also exposes a squash method that retracts the tail pointer to an earlier position.
These extra operations are motivated by their use in an out-of-order processor's reorder buffer, where instructions that are resident in the buffer may need to be squashed upon a branch misprediction.

\subsection{Interface}

The ring buffer exposes nine methods, shown in \Cref{fig:rb-intf}.
Five are value methods: $\mathit{empty}$ and $\mathit{full}$ return the status of the buffer; $\mathit{first}$ returns the element at the head; $\mathit{tail}$ returns the current tail index; and $\mathit{sub}$ returns the element at a given logical index.
Four are action methods: $\mathit{enq}$ inserts an entry at the tail and advances the tail, $\mathit{deq}$ removes the element at the head, $\mathit{squash}$ retracts the tail to a given earlier position discarding pending entries, and $\mathit{upd}$ overwrites an existing entry in place.
Each action method guards against invalid calls: $\mathit{enq}$ aborts when the buffer is full; $\mathit{deq}$ and $\mathit{first}$ abort when it is empty; and $\mathit{sub}$, $\mathit{upd}$, and $\mathit{squash}$ abort when the provided index does not have a valid element there.

\begin{figure}[H]
\begin{minted}{coq}
Definition RingBuffer (@$\varepsilon$@ : type) (N : nat) : ModSig :=
  interface
    method  Bit#(1)        empty;
    method  Bit#(1)        full;
    method  @$\varepsilon$@              first;
    method  Bit#(N)        tail;
    method  Entry          sub     <Bit#(N) idx>;
    method  Action         deq;
    method  Action         enq     <@$\varepsilon$@ e>;
    method  Action         squash  <Bit#(N) tail'>;
    method  Action         upd     <Bit#(N) idx, Entry e>;
  endinterface.
\end{minted}
\caption[Ring-buffer interface]{Interface definition for the ring buffer, parameterized by entry type and address width.}
\label{fig:rb-intf}
\end{figure}

\subsection{Specification}

Since specifying the behavior of pointers requires modular arithmetic, both the specification and the implementation rely on a library of modular arithmetic on natural numbers.
\Cref{fig:rb-nat-ring} summarizes the key definitions.

\begin{figure}[ht]
\begin{minted}[mathescape]{coq}
Definition nat_wrap x  := x mod 2 ^ N.
Definition x @$\oplus$@ y := (x + y) mod 2 ^ N.
Definition x @$\ominus$@ y := (x + 2 ^ N - y mod 2 ^ N) mod 2 ^ N.
Definition x @$\otimes$@ y := (x * y) mod 2 ^ N.
Definition @$\ominus$@ x := (2 ^ N - x mod 2 ^ N) mod 2 ^ N.
Definition x @$\equiv$@ y := nat_wrap x = nat_wrap y.
\end{minted}
\caption[Modular arithmetic over naturals]{Modular arithmetic definitions over natural numbers.}
\label{fig:rb-nat-ring}
\end{figure}

These operations form a commutative ring under the equivalence $x \equiv y$, registered with Rocq's $\mathit{ring}$ tactic, so that goals such as $(x \oplus y) \ominus y \equiv x$ close automatically.
The library also proves that each operation corresponds to its bitvector counterpart under the conversion $\mathit{nat\_to\_bv}$, enabling implementation-level bitvector goals to be translated into specification-level modular-arithmetic goals during refinement proofs.

The specification represents the ring buffer abstractly as an ordered list of entries together with a physical tail index.
\Cref{fig:rb-spec-state} shows the abstract state record.

\begin{figure}[ht]
\begin{minted}[mathescape]{coq}
Record State := mkState {
  entries       : list @$\varepsilon$@;
  tail          : nat;
  size          := length entries;
  head          := tail @$\ominus$@ size;
  wf            : tail < 2 ^ N /\ size <= 2 ^ N;
}.
\end{minted}
\caption[Ring-buffer abstract state]{Abstract state for the ring buffer specification.}
\label{fig:rb-spec-state}
\end{figure}

The abstract model stores the logical contents of the buffer as the list $\mathit{entries}$, ordered from head to tail, along with a $\mathit{tail}$ index.
The $\mathit{head}$ index is derived as $\mathit{tail} \ominus \mathit{size}$, the ring slot that is $\mathit{size}$ positions behind the tail.
A \emph{logical index} is a ring address $j$ that falls within the live portion of the buffer: one satisfying $s.\mathit{valid\_index}(j)$, meaning the forward distance $j \ominus \mathit{head}(s)$ is strictly less than $\mathit{size}(s)$.
The $\mathit{sub}$ and $\mathit{upd}$ operations access elements by logical index, and $\mathit{squash}$ accepts a new tail that must itself be a logical index.
The representation invariant $\mathit{wf}$ bounds $\mathit{tail}$ and $\mathit{size}$ to stay within $2^{N}$.
When the head equals the tail, the buffer may be full or empty; the size disambiguates the two cases, so no explicit full flag appears in the abstract state.
The method specifications are written in the predicate-transformer style described in \Cref{sec:Golemma-spec-framework}.
\Cref{fig:rb-spec-methods} shows the specifications for the two core action methods.

\begin{figure}[ht]
\begin{minted}[escapeinside=||]{coq}
Definition enq' : MRASpec State (mkMetSig <|$\varepsilon$| e> Void) :=
  fun '(args, st) P =>
    exists Hfull : ~ full st,
      P (tt, enq st (args[[@0]]) Hfull).

Definition deq' : MRASpec State (mkMetSig <> Void) :=
  fun '(_, st) P =>
    exists Hempty : ~ empty st,
      P (tt, deq st Hempty).
\end{minted}
\caption[Ring-buffer specifications]{Predicate-transformer specifications for $\mathit{enq}$ and $\mathit{deq}$. The existential witness encodes the precondition as a proof term passed directly to the abstract operation.}
\label{fig:rb-spec-methods}
\end{figure}

The $\mathit{enq}$ operation on the abstract state appends the new element to $\mathit{entries}$ and increments $\mathit{tail}$ by one modulo $2^{N}$.
The $\mathit{deq}$ operation drops the head of $\mathit{entries}$ and leaves $\mathit{tail}$ unchanged, so that subsequent indices shift down by one.
The $\mathit{squash}$ operation truncates $\mathit{entries}$ to the first $\mathit{(index\;st\;tail')}$ elements and sets $\mathit{tail\;:=\;tail'}$.
The $\mathit{upd}$ and $\mathit{sub}$ operations perform a list update and a list lookup, respectively, at the physical offset computed by $\mathit{index}$.

\subsection{Implementation}

The implementation maintains a fixed-size vector of registers for entries and three control registers: $\mathit{head}$ and $\mathit{tail}$ as $N$-bit values and $\mathit{full}$ as a one-bit flag.
\Cref{fig:rb-impl} shows the module declaration.

\begin{figure}[ht]
\begin{minted}{coq}
module mkRingBuffer#(RingBuffer#(@$\varepsilon$@, N));
  Vector#(2 ^ N, Reg#(@$\varepsilon$@)) entries <- replicateM(mkReg());
  Reg#(Bit#(N)) head <- mkReg();
  Reg#(Bit#(N)) tail <- mkReg();
  Reg#(Bit#(1)) full <- mkReg();
\end{minted}
\caption[Ring-buffer implementation]{Module declaration for the ring buffer implementation.}
\label{fig:rb-impl}
\end{figure}

The $\mathit{enq}$ method writes the new element to $\mathit{entries[tail]}$, advances $\mathit{tail}$ by one using bitvector addition (which wraps naturally at $2^N$), and updates $\mathit{full}$ by checking whether the new tail equals $\mathit{head}$.
The validity check for $\mathit{sub}$, $\mathit{upd}$, and $\mathit{squash}$ is computed as the comparison $\mathit{(idx - head) < (tail - head)}$, which effectively checks if $\mathit{entries[idx]}$ is a valid element.
Unlike the specification, which derives the $\mathit{full}$ condition from the size, the implementation must maintain the $\mathit{full}$ flag explicitly to disambiguate whether the queue is empty or full when the head equals the tail.

\subsection{Verification}

The refinement relation $R$ connects implementation states to specification states as shown in \Cref{fig:rb-rel}.
The relation has four components.
First, for every valid logical index $i$ in the specification's ring buffer, the corresponding entry in the implementation's ring buffer matches. The notation $\mathit{xs !!! i}$ denotes a lookup into a list $\mathit{xs}$ at position $\mathit{i}$, returning a default value if the index is out of bounds.
Second, the implementation $\mathit{head}$ and $\mathit{tail}$ registers equal the bitvector representations of the specification's head and tail.
Third, the implementation $\mathit{full}$ bit matches with the specification's full predicate.
Notably, $R$ imposes no constraint on vector slots at indices that are not valid in the specification, so the implementation is allowed to hold arbitrary data in those positions.

\begin{figure}[ht]
\begin{minted}{coq}
Definition R (ist : ImplState) (sst : SpecState) :=
  (forall i, Spec.valid_index sst i ->
     Impl.entries ist !!! i = Spec.sub sst i) /\
  Impl.head ist = nat_to_bv (Spec.head sst) /\
  Impl.tail ist = nat_to_bv (Spec.tail sst) /\
  Impl.full ist = Spec.full sst.
\end{minted}
\caption[Ring-buffer refinement relation]{Refinement relation for the ring buffer. Implementation entries must agree with the specification list at every valid index, with control registers matching under bitvector conversion.}
\label{fig:rb-rel}
\end{figure}

The main theorem is proved by first applying \Cref{def:refines-module}, which produces one proof obligation per method.
We derive a suite of 52 helper lemmas that reason about combinations of methods (e.g. taking the tail of the specification buffer after a dequeue).
We use these helper lemmas to rewrite proof obligations without internally inspecting the components of the specification state.
Reasoning about the implementation requires destructing the state and combining bitvector arithmetic facts with modular arithmetic lemmas.

\begin{lemma} \label{lem:rb-full-tail}
The $\mathit{full}$, $\mathit{head}$ and $\mathit{tail}$ methods each refine their specifications under $R$.
\end{lemma}

\begin{proof}[Proof]
Each follows immediately from $R$, which directly relates the registers in the implementation with their counterparts in the specification.
\end{proof}

\begin{lemma} \label{lem:rb-empty}
The $\mathit{empty}$ method refines its specification under $R$.
\end{lemma}

\begin{proof}[Proof]
Assume $(i, s) \in R$.
The implementation checks $\neg i.\mathit{full} \wedge (i.\mathit{head} = i.\mathit{tail})$.
By $R$, we are given the following:
\[
(i.\mathit{full} = s.\mathit{full}) \land (i.\mathit{head} = s.\mathit{full}) \land (i.\mathit{tail} = s.\mathit{tail}).
\]
A helper lemma establishes that $s.\mathit{head} = s.\mathit{tail} \iff s.\mathit{full} \lor s.\mathit{empty}$.
Therefore the implementation's conjunction reduces to $\neg s.\mathit{full} \wedge (s.\mathit{head} = s.\mathit{tail})$, which holds exactly when $s.\mathit{empty}$.
\end{proof}

\begin{lemma} \label{lem:rb-first}
The $\mathit{first}$ method refines its specification under $R$.
\end{lemma}

\begin{proof}[Proof]
Assume $(i, s) \in R$.
The non-empty guard agrees with the specification's precondition by the same reasoning as \Cref{lem:rb-empty}.
A helper lemma shows that when $s$ is non-empty, $s.\mathit{head}$ is a valid index.
By $R$, we get $i.\mathit{entries}[s.\mathit{head}] = s.\mathit{sub}(s.\mathit{head}) = s.\mathit{first}$.
\end{proof}

\begin{lemma} \label{lem:rb-sub}
The $\mathit{sub}$ method refines its specification under $R$.
\end{lemma}

\begin{proof}[Proof]
Assume $(i, s) \in R$ and logical index $j$.
The implementation's range check computes $(j - i.\mathit{head}) < (i.\mathit{tail} - i.\mathit{head})$ using bitvector arithmetic.
This formula is equivalent to $(j \ominus s.\mathit{head}) < s.\mathit{size}$, which is the definition of $s.\mathit{valid\_index}(j)$.
By $R$, we get $i.\mathit{entries}[j] = s.\mathit{sub}(j)$ given $s.\mathit{valid\_index}(j)$.
\end{proof}

\begin{lemma} \label{lem:rb-enq}
The $\mathit{enq}$ method refines its specification under $R$.
\end{lemma}

\begin{proof}[Proof]
Assume $(i, s) \in R$ and entry $e$.
The fullness check agrees by \Cref{lem:rb-full-tail}.
After writing $i.\mathit{entries}[i.\mathit{tail}] := e$ and advancing $i.\mathit{tail}$ by one, let $i'$ and $s' = s.\mathit{enq}(e)$ be the successor states.
We must show $(i', s') \in R$.
For array agreement, fix a valid index $j$ of $s'$.
If $j$ was already a valid index of $s$, then $i'.\mathit{entries}[j] = i.\mathit{entries}[j]$ and by $R$ we get $i.\mathit{entries}[j] = s.\mathit{sub}(j) = s'.\mathit{sub}(j)$.
Otherwise $j = s.\mathit{tail}$, the newly occupied slot, and $i'.\mathit{entries}[j] = e = s'.\mathit{sub}(j)$.
For the remaining conjuncts, $s'$ updates $\mathit{head}$, $\mathit{tail}$, and $\mathit{full}$ in the same way as $i'$, so those components of $R$ are reestablished.
\end{proof}

\begin{lemma} \label{lem:rb-deq}
The $\mathit{deq}$ method refines its specification under $R$.
\end{lemma}

\begin{proof}[Proof]
Assume $(i, s) \in R$.
The empty check agrees by \Cref{lem:rb-empty}.
After advancing $i.\mathit{head}$ by one and clearing $i.\mathit{full}$, let $i'$ and $s' = s.\mathit{deq}()$ be the successor states.
We must show $(i', s') \in R$.
For array agreement, if $j$ is valid in $s'$ then $j$ was valid in $s$ and $s'.\mathit{sub}(j) = s.\mathit{sub}(j)$; by $R$ we get $i'.\mathit{entries}[j] = i.\mathit{entries}[j] = s'.\mathit{sub}(j)$.
For the head pointer, $s'.\mathit{head} = s.\mathit{head} \oplus 1$, so by $R$ we get $\mathit{nat\_to\_bv}(s'.\mathit{head}) = i.\mathit{head} + 1 = i'.\mathit{head}$.
Dequeue always produces a non-full state, so $i'.\mathit{full} = 0 = s'.\mathit{full}$.
\end{proof}

\begin{lemma} \label{lem:rb-squash}
The $\mathit{squash}$ method refines its specification under $R$.
\end{lemma}

\begin{proof}[Proof]
Assume $(i, s) \in R$ and new tail index $\mathit{tail}'$.
The range check on $\mathit{tail}'$ translates to $s.\mathit{valid\_index}(\mathit{tail}')$ by similar logic to \Cref{lem:rb-sub}.
After setting $i.\mathit{tail} := \mathit{tail}'$ and $i.\mathit{full} := 0$, let $i'$ and $s' = s.\mathit{squash}(\mathit{tail}')$ be the successor states.
We must show $(i', s') \in R$.
For array agreement, if $j$ is valid in $s'$ then $j$ was valid in $s$ and $s'.\mathit{sub}(j) = s.\mathit{sub}(j)$; by $R$ we get $i'.\mathit{entries}[j] = i.\mathit{entries}[j] = s'.\mathit{sub}(j)$.
By construction, $i'.\mathit{tail} = \mathit{tail}'$ and $i'.\mathit{full} = 0$ match $s'$.
\end{proof}

\begin{lemma} \label{lem:rb-upd}
The $\mathit{upd}$ method refines its specification under $R$.
\end{lemma}

\begin{proof}[Proof]
Assume $(i, s) \in R$, logical index $j$, and entry $e$.
The range check translates as in \Cref{lem:rb-sub}.
After writing $i.\mathit{entries}[j] := e$, let $i'$ and $s' = s.\mathit{upd}(j, e)$ be the successor states.
We must show $(i', s') \in R$.
For array agreement, fix a valid index $k$ of $s'$ (equivalently, of $s$, since $\mathit{upd}$ preserves size).
If $k = j$, then $i'.\mathit{entries}[k] = e = s'.\mathit{sub}(j)$.
If $k \neq j$, then $i'.\mathit{entries}[k] = i.\mathit{entries}[k]$ and by $R$ we get $i.\mathit{entries}[k] = s.\mathit{sub}(k) = s'.\mathit{sub}(k)$, since distinct valid indices map to distinct physical slots.
The $\mathit{head}$, $\mathit{tail}$, and $\mathit{full}$ registers are unaffected, so those components of $R$ are preserved.
\end{proof}

\begin{theorem} \label{thm:rb-correct}
The ring buffer implementation refines the ring buffer specification.
\end{theorem}

\begin{proof}[Proof]
Apply \Cref{def:refines-module}.
\Cref{lem:rb-full-tail,lem:rb-empty,lem:rb-first,lem:rb-sub,lem:rb-enq,lem:rb-deq,lem:rb-squash,lem:rb-upd} discharge the per-method obligations.
The reset state satisfies $R$ trivially since both the implementation and specification reset to an empty buffer with $\mathit{head}$, $\mathit{tail}$, and $\mathit{full}$ all equal to zero.
\end{proof}

\section{Resource utilization} \label{sec:eval-resource}

\chapter{Conclusion}

This thesis presented two frameworks for mechanized verification of hardware circuits in Rocq. These frameworks target correctness and timing-security properties needed for a full proof of an out-of-order processor.
\Cref{chap:intro,chap:related} introduced the background and motivation behind this thesis.
\Cref{chap:Slate} introduced Slate and the hardware-constant-time property for timing security; \Cref{chap:Golemma} introduced Golemma and the modular-refinement property for functional correctness; and \Cref{chap:cases} applied both to case studies.
\Cref{chap:evaluation} analyzed the amount of effort required to write proofs using these frameworks and compared the resource utilization between Golemma and Bluespec circuits.

\section{Summary}

The first framework, Slate, is a shallowly embedded language based on guarded state monads that models hardware circuits as deterministic state machines.
Slate supports a proof strategy for hardware constant time, in which security reduces to a per-method simulation argument. Erasure contracts makes the attacker's observation model explicit.
This approach was exercised on a variable-latency multiplier, establishing that its timing behavior is independent of the secret second operand.

The second framework, Golemma, is a deeply embedded hardware-description language whose syntax is inspired by Bluespec SystemVerilog.
Golemma provides a module system, typed interfaces, and a weakest-precondition semantics that supports compositional, per-method correctness proofs via modular refinement.
Extensions to the language include type classes for synthesizing structs, enums, and bitflags and a for-loop construct, allowing specifications to be stated directly over familiar Gallina types.
Functional correctness was demonstrated on a ring buffer whose specification is written using modular arithmetic over naturals.

Together, these contributions provide a solid foundation for scaling towards integrated proofs of correctness and timing security for a full out-of-order processor.

\section{Future work}

The immediate next step is to complete the correctness proofs for the remaining submodules of the out-of-order processor.
The reorder buffer specification and implementation are already complete; the remaining work is to finish its correctness proof.
After that, the renaming logic, the functional units, the physical register file, and the register alias tables each require specifications and correctness proofs.

Given those correctness proofs and specifications, the natural target for a conference paper is a composed correctness and timing-security result for a complete out-of-order processor backend without speculation on memory operations.
Looking further ahead, a natural extension is to augment the processor with a hardware defense against transient-execution attacks and to verify that the defense is sufficient.
Speculative taint tracking~\cite{yu2019} is a promising candidate, as it delays secret-dependent memory accesses until they are known to be non-speculative; adding STT to the Golemma model and proving that the resulting processor satisfies the same erasure-contract obligations would close the gap between the abstract security arguments from prior work and a concrete, synthesizable design.

Finally, a proof for an end-to-end theorem that connects the execution of secure source code to a secure processor be another target for a conference paper.
A similar end-to-end theorem has been explored for a pipelined RISC-V processor in Kami and the Bedrock2 compiler~\cite{erbsen2021}.
This proposed work would compose together prior work on retrofitting the Bedrock2 compiler with constant-time preservation~\cite{conoly2025} and a proof of correctness of a complex out-of-order machine in Golemma.
Connecting those results would yield an end-to-end guarantee that a secret-free source program running on the verified out-of-order processor cannot leak information through timing side channels.
