\begin{abstract}
\addcontentsline{toc}{chapter}{Abstract}
High-performance processors rely on out-of-order execution, an instruction-scheduling algorithm that is both difficult to implement correctly and prone to timing side-channel attacks without principled defenses.
This thesis develops an approach using the Rocq proof assistant for specifying and proving timing-security properties of hardware circuits.
To express hardware circuits in a proof-friendly style, the thesis introduces \emph{Slate}, a shallowly embedded surface language built on guarded state monads.
Slate provides imperative-style notation while preserving direct reduction and equational reasoning in Gallina, enabling compact circuit descriptions and reusable proof structure.
The timing-security framework is developed in a general form and then exercised on two case studies: a variable-latency multiplier and a sequential multi-cycle processor.
In parallel, the thesis uses \emph{GoLemma}, a deeply embedded hardware language in Rocq, to establish functional-correctness results for key out-of-order processor submodules.
Together, these contributions provide a foundation for scaling toward integrated proofs of correctness and timing security for full out-of-order cores.
\end{abstract}

\chapter{Introduction}
This thesis contributes an approach for proving correctness and timing properties for hardware circuits described in the Rocq proof assistant.
This work serves as the first step towards the mechanization of entire correctness and timing proofs for a realistic, modern computer system including a single out-of-order core complete with a memory hierarchy.

\section{Background: out-of-order execution} \label{sec:ooo}
Most modern high-speed processors are based on out-of-order (OoO) execution, which allows the CPU to dynamically reorder instructions based on the availability of functional units (FUs) and input operands, minimizing structural hazards.
Instruction scheduling algorithms supporting out-of-order execution with precise exceptions started with Tomasulo's algorithm~\cite{tomasulo1967}, which first introduced the concept of register renaming to eliminate false dependencies between instructions.
Out-of-order processors combined with another ubiquitous performance optimization, speculative execution, are vulnerable to transient execution attacks, which in-order processors are not.
Principled defenses must be carefully designed to ensure that attackers cannot exploit programs that are safe for in-order processors to execute but unsafe for out-of-order processors.

\Cref{fig:fig1} shows a block diagram for the particular variant of the out-of-order algorithm discussed in this thesis.
The algorithm is a modified design based on the MIPS R10000 (R10K) processor~\cite{r10k1996}.

\subsection{Pipeline overview}
Our algorithm is split into seven separate pipeline stages: fetch, decode, rename, issue, execute, writeback, and retire.
The front end stage fetches instructions from the instruction stream and decodes them into an internal data type the machine can schedule.
The renaming stage then maps architectural registers to a larger physical register file so false dependencies can be dissolved.
The issue stage selects ready renamed operations and sends them to functional units that execute the actual computation.
The writeback stage then posts results to the physical registers so that later instructions can consume operands as they become ready.
The retire stage retires instructions in program order, committing their effects to the architectural state.

\subsection{Renaming, snapshots, and recovery}
Our design diverges from the R10K algorithm in two major aspects to simplify which invariants are needed to write proofs.
The most important difference is how the renaming mechanism works.
The original R10K algorithm records, in the reorder buffer (ROB), the previous physical register index for each instruction's destination so that the rename map can be rolled back precisely when a younger path is squashed.
We instead maintain two register alias tables (RATs), avoiding the need to track a per-instruction previous physical destination in the ROB.
The \emph{future} table holds the most up-to-date speculative mapping from architectural registers to physical registers, while the \emph{commit} table records the mapping as of the last retired instruction.
Squashing speculative state can then be performed by reverting the \emph{future} back to the \emph{commit} table.

Rolling back the alias tables alone would be insufficient, because speculative execution can still overwrite physical register contents before those instructions retire; restoring only the \emph{architectural-to-physical} (arch-to-phys) mapping would leave stale or partially updated values visible under the recovered names.
Our design therefore pairs the dual RAT with explicit snapshots of register \emph{values} (not just of the rename map).
Enough snapshots are retained to reconstruct the physical register file as it stood at the last retired instruction, so a squash restores both the mapping from each architectural register to its physical backing and the bit patterns stored in those physical registers at that point.
The maximum number of snapshots is finite, so it bounds the number of branches the machine can speculate ahead; once that budget is exhausted, the issue stage must stall until retire frees snapshot capacity.

\subsection{Speculative memory}
A second major deviation concerns speculative memory accesses.
To make timing properties easier to state and prove at first, we delay all speculative loads and stores.
This policy is extremely pessimistic: it sacrifices the performance that comes from speculative execution when interacting with memory and is not meant as a proposal for a serious, production-ready design.
Looking ahead, we expect that a more realistic design can be achieved by pairing speculative memory execution with a dynamic taint-tracking defense, together with a mechanized proof that the resulting design still satisfies the same class of timing-security obligations; that extension is left to future work.

\section{Goal: verifying hardware circuits}
This thesis develops techniques for proving correctness and timing properties of synthesizable hardware described in Rocq.
The intended workflow is to state and prove theorems against source-level circuit semantics, then relate them to the semantics of synthesizable Verilog through a verified compiler.
Prior work on security proofs for hardware tends to reason about simplified ISA-level models instead of RTL.
This thesis focuses on synthesizable circuits because they are consumable by downstream synthesis tools for future evaluation on FPGAs or ASICs.

\chapter{Related work}

This chapter first surveys prior work on formal correctness for hardware (\Cref{sec:related-correctness}), which informs the implementation, specification and proof methodologies used throughout the remainder of this thesis.
Then it covers prior work on timing security, both the attacks that motivate this work and the hardware and software defenses which mitigate them (\Cref{sec:related-timing}).

\section{Correctness} \label{sec:related-correctness}

Several prior projects have tackled hardware correctness by embedding hardware description languages directly into a proof assistant.

Kami~\cite{choi2017} is a hardware description language, embedded in Rocq, for specifying and verifying hardware circuits in the style of Bluespec SystemVerilog.
Circuits are semantically treated as labeled transition systems composed of guarded atomic rules, and correctness arguments proceed by refinement between more-abstract and more-concrete specifications.

Ko\^{i}ka~\cite{koika2020} is another language for rule-based hardware design that gives Bluespec-style rules a precise sequential semantics characterized by the \emph{one-rule-at-a-time} (ORAAT) property.
Circuits are described as collections of rules over register state, and the language provides a scheduler that resolves rule conflicts while preserving a one-rule-at-a-time execution model.
Ko\^{i}ka is accompanied by a verified compiler that lowers circuit descriptions to RTL, providing an end-to-end path from high-level specifications to synthesizable netlists.
However, Ko\^{i}ka does not support a module system, so circuits must be described monolithically and lacks compositional reasoning.

\section{Security} \label{sec:related-timing}

The Meltdown~\cite{lipp2018} and Spectre~\cite{kocher2019} attacks, disclosed in 2018, demonstrated that speculative execution, while critical for performance, introduces new side-channel vulnerabilities absent in in-order processors.
Both attacks exploit the fact that transiently executed instructions leave measurable traces in microarchitectural state, such as cache-residency patterns, that an attacker can recover.
These attacks motivated a large body of research~\cite{koruyeh2018,maisuradze2018,kiriansky2018,schwarz2019,minkin2019} on transient execution attacks.

Hardware-software contracts~\cite{guarnieri2021} formalize hardware defenses by specifying, for a particular design, both what information is leaked and which speculative paths are explored.
An analysis of an abstract out-of-order processor protected by speculative taint tracking~\cite{yu2019} shows that the defense satisfies a contract in which leakage events are branch targets, addresses of loads and stores, and operands of variable-latency instructions.
The key limitation of this line of work is that it reasons about abstract processor models rather than concrete designs expressible in a hardware description language; this thesis targets the gap to synthesizable RTL.

At the software level, constant-time verification for nondeterministic programs and compilers~\cite{conoly2025} develops techniques for proving constant-time properties in the Rocq proof assistant.
Using a formulation of constant time as a single-copy property over omnisemantics, the work proves end-to-end preservation of constant-time routines compiled from Bedrock2~\cite{erbsen2021} source to RISC-V machine code.
The single-copy formulation of constant time used in this thesis is taken from that work, repurposing the approach from software compilers to hardware verification.

SecSep~\cite{song2025} addresses the gap between source-level constant-time guarantees and speculative execution on real hardware through an assembly-level transformation framework.
SecSep targets processors similar to ProSpeCT~\cite{daniel2023}, which partitions memory into regions whose speculative loads are delayed until non-speculative and regions whose loads are not, provably enforcing the constant-time policy without relying on fencing.
SecSep rewrites compiled assembly programs so that secret and public data are partitioned on the stack, using a new typed assembly language called \emph{Octal} to guide the transformation.
The approach correctly handles secrets on the stack and achieves a low additional runtime overhead on cryptographic programs.

Prior work on hardware timing security~\cite{duxovni2018} builds off the Kami framework to reason about timing-security properties of hardware circuits, showing that the same embedding used to verify functional correctness can also support security arguments.
More recent work~\cite{lau2024} formally specifies and verifies strong timing isolation for hardware enclaves, reasoning about cycle-accurate circuit semantics with respect to an air-gapped-machine formulation of isolation and applying the method to a multicore, pipelined RISC-V design.
This thesis continues in the same tradition, carrying these timing-security proof techniques into the setting of individual submodules of a large out-of-order machine.

\section{State machines} \label{sec:timing-state-machine}

A \emph{state machine} is an abstraction used to describe the behavior of a computation by defining its possible states and the rules for transitioning between them.
Hardware modules function as physical state machines: registers serve as the memory that holds the current state, while combinational logic describes the transition function that determine both the module's outputs and its next state as a function of the current configuration.
We view hardware modules as interacting through well-defined interfaces composed of methods rather than raw, unstructured signals.
Each discrete state transition in the system corresponds to the execution of a method, supplied with arguments that represent the transition's data.
This approach is consistent with prior work~\cite{koika2020,choi2017}, which views hardware modules as high-level transition systems whose transitions lie on external method boundaries.
We adopt the following formal definition of a state machine.

\begin{definition}[State machine]
  A state machine $\mathcal{M}$ is a 5-tuple $(S, s_0, I, O, \delta)$ where $S$ is a set of states, $s_0 \in S$ is the initial state, $I$ is a set of inputs, $O$ is a set of outputs, and
  \[
    \delta : S \times I \rightharpoonup S \times O
  \]
  is a partial transition function.
  Here $\xrightharpoonup{}$ denotes that certain methods can abort and produce no state change or output.
\end{definition}

We model hardware circuits as such machines.
A state $s \in S$ captures the current values of the sequential logic relevant to the module.
Each input in $I$ represents a \emph{method invocation}: a variant type carrying a method tag together with the argument values for that call.
The transition function $\delta$ is partial on $S \times I$ to reflect that the model may not produce a well-defined successor state and output when the call is \emph{aborted} by a guard (for example, dequeuing from an empty buffer).
An aborted call leaves the circuit in the same configuration.
This convention simplifies trace definitions and simulation proofs: each input sequence determines one run, and the optional output indicates whether the call produced a visible event in the output trace.

For the purpose of reasoning about security with this framework, we additionally restrict attention to circuits whose behavior is \emph{deterministic} in the sense above: for each $s \in S$ and $\iota \in I$, there is at most one pair $(s', o) = \delta(s,\iota)$ with $o \in O$.
This restriction is convenient for reasoning about the proof obligations in \Cref{sec:timing-ct} and \Cref{sec:timing-proof}, where we compare runs under related inputs and want equality of traces without existentially quantifying over implementation-defined resolution of nondeterminism in proofs.
Determinism is a design choice of convenience, not necessity; as discussed in \Cref{chap:golemma}, the framework for deeply embedded circuits can use nondeterministic semantics (in particular, omnisemantics~\cite{chargueraud2023}) to describe specifications.

Given an input sequence $\iota_1, \iota_2, \ldots, \iota_n \in I$, repeatedly applying $\delta$ from the initial state $s_0$ defines a \emph{run} of the machine.
The corresponding sequence of per-step observations is called the machine's \emph{trace}.
Concretely, the machine $\mathcal{M}$ produces a sequence of optional outputs by unfolding $\delta$ from $s_0$:
\[
  s_0 \xrightarrow{\iota_1 / o_1} s_1 \xrightarrow{\iota_2 / o_2} s_2 \;\cdots\; \xrightarrow{\iota_n / o_n} s_n, \qquad o_i \in O,\ s_i \in S.
\]
The \emph{observation trace} of the run is the list
\[
  \mathcal{T}\bigl( \mathcal{M}, \vec\iota \bigr) \;\triangleq\; o_1,\ldots,o_n
.\]
For timing security, an adversary interprets elements of $O$ (modulo erasure, described in \Cref{sec:timing-ct}) as leaked information.
Aborted transitions do not produce outputs in the observation trace.

The state machine model is mechanized in Rocq as a plain record (\Cref{fig:circuit-typeclass}), parameterized by a state type~\texttt{s}, an input type~\texttt{i}, and an output type~\texttt{o}.
These parameters correspond directly to $S$, $I$, and $O$.

\begin{figure}[H]
\begin{minted}{coq}
Definition Action s t := s -> option (s * t).

Record StateMachine (s i o : Type) := {
  state_machine_rst  : s;
  state_machine_call : i -> Action s o;
}.
\end{minted}
\caption{The \texttt{StateMachine} record. The fields \texttt{state\_machine\_rst}
  and \texttt{state\_machine\_call} correspond to $s_0$ and to $\delta$ respectively.}
\label{fig:circuit-typeclass}
\end{figure}

\section{Encoding circuits} \label{sec:timing-monads}

This section introduces Slate, a language for describing hardware as executable descriptions in Rocq.
It first compares shallow and deep embeddings and explains why Slate is implemented as a shallow embedding (\Cref{sec:timing-monads-shallow}).
It then describes state monads and guarded state monad (actions) and motivates them as a way to model imperative programs shallowly in Rocq (\Cref{sec:timing-monads-state}).
Finally, it presents the grammar of Slate and a small illustrative example (\Cref{sec:timing-monads-slate}).

\subsection{Shallow and deep embeddings} \label{sec:timing-monads-shallow}
A \emph{shallow embedding} encodes an object language directly as terms of the host language.
In this style, the host's type system, reduction rules, and terms \emph{are} the semantics of the object language.
In Rocq, that means method bodies are ordinary Gallina functions rather than abstract syntax trees requiring a separate interpreter.
By contrast, a \emph{deep embedding} defines explicit syntax via inductive datatypes for commands and expressions.
A deep embedding must also include a separate interpreter or a relationally defined operational semantics to be useful for verification work.

Each style has distinct advantages and costs.
Shallow embeddings are typically easier for verification work, because they directly leverage Rocq's facilities for Gallina; many proof steps reduce by unfolding and simplification rather than by reasoning about an interpreter or operational semantics.
The main drawback is that source-level program transformations, such as compiler passes, are tricky to implement, since syntax is not explicit.
These tasks often require reification and metaprogramming support to recover a syntax tree.

Deep embeddings make syntax explicit from the beginning, so compiler construction and optimization passes are easier to express.
However, they impose additional verification overhead: one must define and reason about an interpreter (or step relation), and prove that this semantics aligns with the intended meaning.
As a result, proofs on deep embeddings are more involved for similar theorems on shallow embeddings.
\emph{Slate}, the surface language used for timing-security verification in this thesis, is a shallow embedding.
Its core semantics work as an imperative language with mutable references.
We introduce the state monad structure that underlies Slate method definitions.

\subsection{State monads and actions} \label{sec:timing-monads-state}
\paragraph{State monads}
A \emph{state monad} over state type $s$ and result type $t$ is defined as
\[
  \mathsf{State}\;(s, t) \;\triangleq\; s \to (s * t)
,\]
that is, a function that takes an input state and returns a new state together with a result.
A stateful computation can be understood as a function that consumes a state and returns a pair of the updated state and a result value.
This representation is powerful because it expresses side effects sequentially by chaining computations with \mintinline{coq}{bind}.
Mutable references do not need to be primitively supported by the meta-language (i.e. Gallina); instead, they are ordinary functions that transform an explicit state argument.
As a consequence, one can describe sequential hardware behavior in a pure functional setting and still recover the usual intuition of sequential logic, by reading registers at the start of the cycle, performing some combinational function on those values, and updating the registers at the end of the cycle.

\Cref{fig:state-monad-core} shows the state monad encoded in Rocq together with its basic operations.
The constructor \mintinline{coq}{ret} lifts a pure value into the monad without changing the state, while \mintinline{coq}{bind} sequences two computations by feeding the result and resulting state of the first computation into the second.
The infix operator \mintinline{coq}{>>=} is another way to express \mintinline{coq}{bind}.
State monads also provide state-specific primitives, notably \mintinline{coq}{get} and \mintinline{coq}{put}: \mintinline{coq}{get} returns the current state as a value, and \mintinline{coq}{put} replaces the current state with a new one.

\begin{figure}[H]
\begin{minted}{coq}
Definition State s t := s -> (s * t).

Definition ret {s t} (x : t) : State s t :=
  fun st => (st, x).

Definition bind {s t1 t2} (m : State s t1) (f : t1 -> State s t2) :=
  fun st =>
    let '(st', x) := m st in
    f x st'.

Definition get {s} : State s s :=
  fun st => (st, st).

Definition put {s} (x : s) : State s unit :=
  fun _ => (x, tt).
\end{minted}
\caption{State monad definitions in Rocq.}
\label{fig:state-monad-core}
\end{figure}

\paragraph{Guarded state monads}
A \emph{guarded state monad}, also referred to as an \emph{action}, over state type $s$ and result type $t$ is the type of computations
\[
  \mathsf{Action}\;(s,t) \;\triangleq\; s \to \texttt{option}\;(s * t)
,\]
that is, a function that takes an input state and either returns a new state together with a result or aborts.
The framework uses this to model method calls that are enabled only when their guards hold.
The \mintinline{coq}{Some} case represents a successful step that returns an updated state and a value.
The \mintinline{coq}{None} case represents an aborted step (for example, when a guard is false).
\Cref{fig:guarded-state-monad-core} shows the guarded state monad encoded in Rocq together with its basic operations.
\mintinline{coq}{ret}, \mintinline{coq}{bind}, \mintinline{coq}{get}, and \mintinline{coq}{put} work similarly to their state monad analogues.

\begin{figure}[H]
\begin{minted}{coq}
Definition Action s t := s -> option (s * t).

Definition ret {s t} (x : t) : Action s t :=
  fun st => Some (st, x).

Definition bind {s t1 t2} (m : Action s t1) (f : t1 -> Action s t2) :=
  fun r =>
    match m r with
    | Some (s, x) => f x s
    | None => None
    end.

Definition get {s} : Action s s :=
  fun st => Some (st, st).

Definition put {s} (x : s) : Action s unit :=
  fun _ => Some (x, tt).

Definition modify {s} (f : s -> s) : Action s unit :=
  fun st => Some (f st, tt).

Definition fail {s t} : Action s t :=
  fun _ => None.
\end{minted}
\caption{Guarded state monad definitions in Rocq.}
\label{fig:guarded-state-monad-core}
\end{figure}

\paragraph{Modify}
\mintinline{coq}{modify f} updates the state by applying a function \mintinline{coq}{f}.
It can be used as a derived write primitive: read the current state, transform it, then commit the result.
In hardware, this operation is a natural fit for register updates where the next state is expressed as a pure function of the current state.
Proof obligations involving \mintinline{coq}{modify} usually reduce to equalities between functions applied to the same input state.

\paragraph{Guard failure}
\mintinline{coq}{fail} returns \mintinline{coq}{None} for any input state.
In hardware description semantics, this represents an aborted method call (for example, violating a ready condition).
At the state machine level, this is later normalized into a step with no observable output and no state changes.
Introducing failure in the monad itself keeps reasoning about guards local to method definitions.

\paragraph{Submodule calls}
The operator \mintinline{coq}{call} implements modular method calls over nested state.
It uses \mintinline{coq}{getter} to project a substate, runs an \mintinline{coq}{Action} on that substate, and then re-integrates the updated substate using \mintinline{coq}{setter} as shown in \Cref{fig:action-call}.
If the callee action fails, the caller fails.
This mirrors calling semantics in Bluespec SystemVerilog, where a method call that fails to fire causes the caller to fail to fire.

\begin{figure}[H]
\begin{minted}{coq}
Definition call {s a b}
  (getter : s -> a)
  (setter : a -> s -> s)
  (action : Action a b) : Action s b :=
  fun st =>
    fun _ =>
      match action (getter st) with
      | Some (st', x) => Some (setter st' st, x)
      | None => None
      end).
\end{minted}
\caption[Call operator for Actions]{\texttt{call} operator lifts the effects of submodule action to a larger state.}
\label{fig:action-call}
\end{figure}

\subsection{Syntax} \label{sec:timing-monads-slate}
The syntax for Slate is implemented using the notation facilities in Rocq.
Crucially, this notation elaborates directly into actions as described above; it does not introduce a syntax tree.
An abstract syntax of the grammar is given in \Cref{fig:action-grammar}.
The terminal \texttt{term} in the grammar refers to arbitrary Gallina terms, which you can embed directly into Slate.

\begin{figure}[H]
\begin{mdframed}
\[
\begin{array}{rcl}
\text{Expressions} \quad e & ::= & \mathit{term} \mid \texttt{`}\mathit{reg}\texttt{`read()} \\
\\
\text{Statements} \quad s & ::= & \texttt{pass} \mid \texttt{abort} \mid \texttt{return}\ e \mid \texttt{`}\mathit{reg}\texttt{`write}(e) \\
& & \mid \texttt{if}\ (e)\ s\ \texttt{else}\ s \mid \texttt{when}\ (e)\ s \\ & & \mid \texttt{let}\ \mathit{pat}\ \texttt{=}\ e \mid \texttt{begin}\ b\ \texttt{end} \mid \texttt{call}\ \texttt{`}\mathit{submodule}\texttt{`}(e) \\
\\
\text{Blocks} \quad b & ::= & s \mid s\ \texttt{;}\ b \\
\\
\text{Actions} \quad a & ::= & \texttt{begin}\ b\ \texttt{end}
\end{array}
\]
\end{mdframed}
\caption{Abstract syntax of the Slate surface language.}
\label{fig:action-grammar}
\end{figure}

Consider \Cref{fig:empty-surface}, which implements the body of a method called \mintinline{coq}{mark_busy} over the state type \mintinline{coq}{State}.
In Slate, the method reads the phase register into the let binding \mintinline{coq}{ph}, checks the condition \mintinline{coq}{ph == Empty}, writes to the phase register, and returns a Boolean result.
This style is close to imperative code while still denoting a pure Gallina term.
The same method can be written directly with monadic operators (\Cref{fig:empty-bind}).
This version makes sequencing explicit through \mintinline{coq}{>>=}, but is less readable for longer method bodies.
In practice, both forms are definitionally equivalent after notation expansion.


\begin{figure}[H]
\begin{minted}{coq}
Variant Phase := Busy | Empty | Ready.

Definition State := { phase: Phase }.

Definition mark_busy : Action State bool :=
  begin
    let ph = `phase`read();
    if (ph == Empty) begin
      `phase`write(Busy);
      return true
    end else begin
      return false
    end
  end.

\section{Security definitions} \label{sec:timing-ct}

This section introduces the concept of hardware constant time and motivates it as a robust security property that can model a range of attackers.
It first describes the cryptographic constant time security property (\Cref{sec:cct}) and its application to software programs. Then, it extends this property to hardware circuits as the hardware constant time security property (\Cref{sec:hct}).

\subsection{Cryptographic constant time} \label{sec:cct}
Cryptographic software often cares about the \emph{cryptographic constant-time} (CCT) property: attackers should not observe leakage events such as branch direction, memory-access patterns, or variable-latency instructions influenced by secret inputs.
This principle has been formalized and mechanically verified for software programs in multiple prior works~\cite{almeida2016,conoly2025,guarnieri2021}.
\Cref{fig:cct-intuition} illustrates the key intuition with two small routines: one has a secret-dependent branch and is not constant time, while the other computes through bit masking and avoids control-flow dependent on secret data.

More formally, constant-time security is stated as an observational equality under varying secrets: if two executions agree on public inputs, then they must be indistinguishable with respect to the attacker-observable trace.
The next definition makes this precise in the traditional two-copy (hyperproperty) style, before introducing an equivalent one-copy formulation.

\begin{figure}[H]
\begin{minipage}[t]{0.48\linewidth}
\begin{minted}{c}
// Not constant-time
uint32_t f_bad(x, y, secret) {
  uint32_t x;
  uint32_t y;
  uint8_t secret;

  if (secret) return x;
  else return y;
}
\end{minted}
\end{minipage}\hfill
\begin{minipage}[t]{0.48\linewidth}
\begin{minted}{c}
// Constant-time
uint32_t f_ct(x, y, secret) {
  uint32_t x;
  uint32_t y;
  uint8_t secret;

  uint32_t m = 0u - (secret & 1u);
  return (x & m) | (y & ~m);
}
\end{minted}
\end{minipage}
\caption{Constant time example: secret-dependent branch (left) versus branch-free masked selection (right).}
\label{fig:cct-intuition}
\end{figure}

\begin{definition}[Cryptographic constant time as a two-copy hyperproperty] \label{def:two-copy}
A state machine $\mathcal{M} = (S, s_0, I, O, \delta)$ is \emph{cryptographic constant time} (CCT)
if given partitions
\[
  s = (s^{\mathrm{pub}}, s^{\mathrm{sec}}), \quad
  i = (i^{\mathrm{pub}}, i^{\mathrm{sec}}), \quad
  o = (o^{\mathrm{pub}}, o^{\mathrm{sec}})
\]
of states, inputs, and outputs into public and secret components, such that for any two
states $s_1, s_2 \in S$ with $s_1^{\mathrm{pub}} = s_2^{\mathrm{pub}}$, and for any two
input sequences $(i_1, i_2, \ldots)$ and $(i_1', i_2', \ldots)$ in $I$ with $i_k^{\mathrm{pub}} = i_k^{\prime\,\mathrm{pub}},$
the public output components agree at every step $k$, $o_k^{\mathrm{pub}} = o_k^{\prime\,\mathrm{pub}}.$
\end{definition}

This definition is relational: it compares two universally-quantified executions and is therefore a hyperproperty.
Next, we introduce a one-copy formalism of constant time which is logically equivalent to \Cref{def:two-copy}.

\begin{definition}[Cryptographic constant time as a one-copy property]
A state machine $\mathcal{M} = (S, s_0, I, O, \delta)$ is \emph{cryptographic constant time} (CCT)
if given partitions of states, inputs, and outputs
\[
  s = (s^{\mathrm{pub}}, s^{\mathrm{sec}}), \quad
  i = (i^{\mathrm{pub}}, i^{\mathrm{sec}}), \quad
  o = (o^{\mathrm{pub}}, o^{\mathrm{sec}})
,\]
there exists a predictor function
\[
  f_\textsf{predict}
  :
  S^{\mathrm{pub}} \times (I^{\mathrm{pub}})^* \to (O^{\mathrm{pub}})^*
,\]
where $\mathsf{pub\_inputs}(\vec{i})$ is the elementwise projection of an input sequence
to public components and $\mathsf{pub\_outputs}(\mathcal{M}, s_0, \vec{i})$ is the corresponding projected observation trace.
Then, for every initial state $s_0 \in S$ and input sequence $\vec{i} \in I^*$,
\[
  \mathsf{pub\_outputs}(\mathcal{M},s_0,\vec{i})
  =
  f_\textsf{predict}\bigl(s_0^{\mathrm{pub}},\mathsf{pub\_inputs}(\vec{i})\bigr).
\]
\end{definition}

This definition originally comes from work on retrofitting the Bedrock2 compiler to prove preservation of cryptographic constant time across compiler passes~\cite{conoly2025}.
Intuitively, the observable behavior is fully determined by the public projection of the input and the initial state.
This is a one-copy specification style: each run is characterized independently without needing to relate two separate sequences.
This definition is useful for certain proof techniques that can existentially construct a nice predictor function ($f_\textsf{predict}$).
The hardware development in this thesis uses the same one-copy perspective, but restricted to state machine traces with explicit input/output erasure functions that capture the notion of the partitioning state into secret and public components.

\subsection{Hardware constant time} \label{sec:hct}
Hardware constant time is motivated by hardware-software contracts~\cite{guarnieri2021}: the security guarantee is stated relative to an explicit attacker observation model. In that line of work, the contract ties together what information the attacker can observe (leakage events) and what information the system must hide as a function of secret inputs. In this thesis, we adopt the same pattern: we first make the attacker's observational capabilities explicit via erasure contracts (\Cref{def:erasure-spec}), and then define hardware constant time as an invariance of the erased observation when secret (erased) inputs vary.

\begin{definition}[Erasure contract] \label{def:erasure-spec}
For a state machine $\mathcal{M}=(S,s_0,I,O,\delta)$, an \emph{erasure contract} is a pair $\mathcal{E} = (\mathsf{erase_{in}}, \mathsf{erase_{out}})$ where $\mathsf{erase_{in}}$ is a function which erases secret components of inputs $\iota \in I$ and $\mathsf{erase_{out}}$ is a function which erases secret components of outputs $o \in O$ from the observational trace.
\end{definition}

\begin{definition}[Erased observation trace] \label{def:erased-run-outputs}
Given $\mathcal{M}$, $\mathcal{E}$, and an input sequence $\vec\iota \in I^*$, we define
$\mathcal{T}_{\mathcal{E}}\bigl( \mathcal{M}, \vec\iota \bigr)$ as the \emph{erased observation trace} which is the observation trace $\mathcal{T}\bigl( \mathcal{M}, \vec\iota \bigr)$ with private output components deleted.
\end{definition}

\begin{definition}[Erased input sequence] \label{def:erased-run-outputs}
Given $\mathcal{M}$, $\mathcal{E}$, and an input sequence $\vec\iota \in I^*$, we define
$\vec\iota_\mathcal{E}$ as the \emph{erased input sequence} which is the input sequence with private input components deleted.
\end{definition}

\begin{definition}[Hardware constant time] \label{def:hct-single}
A state machine $\mathcal{M}$ is \emph{hardware constant time} (HCT) with respect to an erasure contract $\mathcal{E}$ when
\[
  \forall \vec\iota \in I^*.\;\;
  \mathcal{T}_\mathcal{E}(\mathcal{M}, \vec\iota)
  =
  \mathcal{T}_\mathcal{E}(\mathcal{M}, \vec\iota_\mathcal{E}).
\]
\end{definition}

An erasure contract can be interpreted as a threat model, a description of the observational capabilities of an attacker.
It states which wires of a module's interface are observable to an attacker and which parts are treated as secret.
Concretely, $\mathsf{erase_{in}}$ controls observability on input wires: it preserves attacker-visible components and erases the rest.
Similarly, $\mathsf{erase_{out}}$ controls observability on output wires: it keeps attacker-visible outputs and removes or abstracts outputs that are outside the attacker's view.
Under this interpretation, HCT says that varying secret (erased) inputs cannot change the attacker-visible output trace.

\section{Proof strategy} \label{sec:timing-proof}

This section develops the simulation-based proof strategy used to establish hardware constant time from \Cref{sec:hct}. It first introduces lockstep simulation and its two key variants: self-simulation of a single machine and cross-simulation between an implementation machine and an abstract machine (\Cref{sec:timing-proof-lockstep}). It then derives the trace-inclusion lemmas needed to imply hardware constant time from both simulation strategies (\Cref{sec:timing-proof-trace}).

\subsection{Lockstep simulation} \label{sec:timing-proof-lockstep}
The hardware constant time property (\Cref{def:hct-single}) reasons about entire runs of state machines rather than isolated steps.
This affords flexibility in potential proof techniques that may not reason about each step in isolation, but in the future could allow simulation-based proofs which rely on skipping steps or taking multiple steps.
We will restrict ourselves to only reason about \emph{lockstep} simulations, which disallow skipping or multiple steps.
\Cref{fig:sim-machines} demonstrate the two different proof techniques introduced in this section: first, a machine compared against itself under erased inputs (self-simulation), and second, an implementation machine compared against an abstract specification (cross-simulation).

\paragraph{Setup}
Fix state machines $\mathcal{M}_1$ and $\mathcal{M}_2$ and erasure contract $\mathcal{E}$ as the following
\begin{gather*}
  \mathcal{M}_1 = (S_1, s_{0,1}, I, O, \delta_1), \qquad
  \mathcal{M}_2 = (S_2, s_{0,2}, I, O, \delta_2),\\
  \mathcal{E} = (\mathsf{erase_{in}},\mathsf{erase_{out}}).
\end{gather*}
Fix relation $R \subseteq S_1 \times S_2$. Let $a_1$ and $a_2$ be actions over $S_1$ and $S_2$, respectively, sharing output type $O$.
For related starting states $(s_1,s_2)\in R$, suppose each action steps to a successor state and output:
\[
  a_1(s_1)=(s_1',o_1),\qquad a_2(s_2)=(s_2',o_2).
\]

\begin{definition}[Erased-result relation] \label{def:erased-relation}
Given a relation $R \subseteq S_1 \times S_2$ and erasure contract $\mathcal{E}$, we say $R_\mathcal{E} \subseteq S_1 \times O \times S_2 \times O$ is the \emph{erased-result relation} when
\[
  R_\mathcal{E}\bigl(s_1,o_1,s_2,o_2\bigr)
  \;\triangleq\;
  \mathsf{erase_{out}}(o_1)=\mathsf{erase_{out}}(o_2) \; \land \; R(s_1,s_2).
\]
\end{definition}

\begin{definition}[Action simulation] \label{def:action-refinement}
We say \(a_1\) \emph{simulates} \(a_2\) under \((R,\mathcal{E})\) at \((s_1,s_2)\) when
\[
  R(s_1,s_2) \Rightarrow R_\mathcal{E}\bigl(s_1',o_1,s_2',o_2\bigr).
\]
This condition is depicted as a commutative diagram in~\Cref{fig:sim-action-square}.
\end{definition}

\begin{figure}[H]
\centering
\begin{tikzcd}[column sep=8em,row sep=huge]
  s_1 \arrow[r,"a_1"] \arrow[d,dashrightarrow,"R"'] & (s_1',o_1) \arrow[d,dashrightarrow,"R_\mathcal{E}"']
    \\
  s_2 \arrow[r,"a_2"'] & (s_2',o_2)
\end{tikzcd}
\caption[Action refinement]{Action refinement for \(a_1\) and \(a_2\) as a commutative diagram.}
\label{fig:sim-action-square}
\end{figure}

Next, machine simulation instantiates~\Cref{def:action-refinement} on every method invocation $\iota \in I$ where the abstraction machine observes~\(\mathsf{erase_{in}}(\iota)\) wherever the implementation machine observes~\(\iota\).

\begin{definition}[Machine simulation] \label{def:machine-refinement}
$\mathcal{M}_1$ \emph{simulates} $\mathcal{M}_2$ \emph{under} $(R, \mathcal{E})$ when, for every input $\iota \in I$, all $(s_1,s_2)\in R$, and actions $a_1$ and $a_2$:
\[
a_1 = \lambda s.\; \delta_1(s,\iota)
,\quad
a_2 = \lambda s.\; \delta_2(s,\mathsf{erase_{in}}(\iota))
,\]
$a_1$ simulates $a_2$ under $(R,\mathcal{E})$ at $(s_1,s_2)$ in the sense of~\Cref{def:action-refinement}.
\end{definition}

\paragraph{Self-simulation and cross-simulation}
The self-simulation and cross-simulation instances of this definition are shown in~\Cref{fig:sim-machines}. In self-simulation, the same machine appears on both sides: one side consumes the original input \(\iota\), while the other consumes the erased input \(\mathsf{erase_{in}}(\iota)\). This directly matches the shape of hardware constant time, where we compare a machine against itself under secret variation. In cross-simulation, the left side is a concrete implementation and the right side is a simpler abstract specification that already enforces the intended security behavior; the proof then factors through simulation to transfer that behavior back to the implementation.

\subsection{Trace-equivalence lemmas} \label{sec:timing-proof-trace}
This subsection turns the stepwise lockstep obligations from~\Cref{fig:sim-action-square,fig:sim-machines} into trace consequences. We first define output-erased trace equivalence and show that machine simulation lifts to equality of erased traces. We then use these lemmas to derive the two proof patterns for hardware constant time: direct self-simulation and cross-simulation via a trace-equivalent specification.

\begin{definition}[Erased-trace equivalence] \label{def:outputs-erased-bridge}
Call~\(\mathcal M_2\) \emph{erased-trace equivalent} to~\(\mathcal M_1\) under~\(\mathcal E\) when, for every~\(\vec\iota\in I^*\), the erased-input histories~\(\vec\iota_{\mathcal E}\) induce the same erased observation trace:
\[
  \mathcal{T}^{\mathcal E}(\mathcal M_2,\vec\iota_{\mathcal E})
  =
  \mathcal{T}^{\mathcal E}(\mathcal M_1,\vec\iota_{\mathcal E}).
\]
\end{definition}

\begin{lemma}[Simulation lifts to erased traces]
\label{lem:sim-lift-erased-traces}
If \(\mathcal{M}_1\) refines \(\mathcal{M}_2\) under~\((\mathcal E,R)\) in the sense of~\Cref{def:machine-refinement} and \(R(s_{0,1},s_{0,2})\), then for every input history~\(\vec\iota\in I^*\),
\[
  \mathcal{T}_{\mathcal E}(\mathcal M_1,\vec\iota)
  =
  \mathcal{T}_{\mathcal E}(\mathcal M_2,\vec\iota_{\mathcal E}).
\]
\end{lemma}

\begin{proof}[Proof sketch]
Induct on~\(\vec\iota\); the base case on the empty list is immediate. The inductive case first applies~\Cref{def:machine-refinement} to the head of the trace before appealing to the induction hypothesis on the tails.
\end{proof}

\begin{lemma}[Self-simulation implies hardware constant time]
\label{lem:self-sim-hct}
If~\(\mathcal M\) simulates itself under~\((R,\mathcal E)\) and \(R(s_0,s_0)\), then~\(\mathcal M\) satisfies hardware constant time (\Cref{def:hct-single}) with respect to~\(\mathcal E\).
\end{lemma}

\begin{proof}[Proof sketch]
Instantiate~\Cref{lem:sim-lift-erased-traces} with the specification and implementation both taken to be~\(\mathcal M\); the resulting equation is~\Cref{def:hct-single}.
\end{proof}

\begin{lemma}[Cross-simulation with trace equivalence implies hardware constant time]
\label{lem:cross-sim-hct}
Suppose \(\mathcal{M}_{\mathsf I}\) simulates \(\mathcal{M}_{\mathsf S}\) under~\((R,\mathcal E)\), that \(R(s_{0,\mathsf I},s_{0,\mathsf S})\), and that~\(\mathcal{M}_{\mathsf S}\) is erased-trace equivalent to~\(\mathcal{M}_{\mathsf I}\), as stipulated in~\Cref{def:outputs-erased-bridge}.
Then \(\mathcal{M}_{\mathsf I}\) satisfies hardware constant time (\Cref{def:hct-single}) with respect to~\(\mathcal E\).
\end{lemma}

\begin{proof}[Proof sketch]
Compose~\Cref{lem:sim-lift-erased-traces}, which aligns the~\(\mathcal{M}_{\mathsf I}\) trace against the~\(\mathcal{M}_{\mathsf S}\) trace once inputs are~\(\mathcal E\)-obscured, with erased-trace equivalence (\Cref{def:outputs-erased-bridge}), which rewrites obscured~\(\mathcal{M}_{\mathsf S}\)-behavior onto obscured~\(\mathcal{M}_{\mathsf I}\)-behavior, yielding the self-comparison required by~\Cref{def:hct-single}.
\end{proof}

\subsection{Example proof}
As a concrete example, consider the following one-method machine whose output alternates every cycle its method is called.

\newpage

\begin{minted}[escapeinside=||,mathescape=true]{coq}
Record S := mkS { phase : bool }.

Definition tick : Action S bool :=
  begin
    let p = `phase`read();
    `phase`write(negb p);
    return (negb p)
  end.

Definition |$\mathcal M_{\mathsf{toggle}}$| : StateMachine S unit bool :=
  mkStateMachine (mkS false) (fun '() => tick).

Definition |$\mathsf{erase_{in}}$| := id.
Definition |$\mathsf{erase_{out}}$| := id.
Definition |$\mathcal{E}_{\mathsf{toggle}}$| : ErasureContract I O :=
  (erase_in_toggle, erase_out_toggle).
\end{minted}

\begin{theorem}[Toggle machine is hardware constant time]
\label{thm:toggle-hct}
The machine \(\mathcal M_{\mathsf{toggle}}\) is hardware constant time with respect to \(\mathcal E_{\mathsf{toggle}}\).
\end{theorem}

\begin{proof}[Proof sketch]
Use~\Cref{lem:self-sim-hct} with self-simulation. Let \(R\) be equality on states (equivalently, equality of \mintinline{coq}{phase}). We must show one-step simulation under erased inputs/outputs. Since \(\mathsf{erase_{in}}\) is identity, both sides execute the same method call \mintinline{coq}{Tick}. Since \(\mathsf{erase_{out}}\) is also identity, erased-output agreement reduces to plain output equality.

Now unfold one step of \mintinline{coq}{tick}: both sides read the same current \mintinline{coq}{phase}, both return \mintinline{coq}{negb p}, and both update to \mintinline{coq}{phase := negb p}. Hence outputs are equal and successor states remain related by \(R\). Therefore \(M_{\mathsf{toggle}}\) self-simulates under \((R,\mathcal E_{\mathsf{toggle}})\), and the initial states are trivially related. Applying~\Cref{lem:self-sim-hct} yields hardware constant time.
\end{proof}
