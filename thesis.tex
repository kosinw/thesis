\begin{abstract}
High-performance processors rely on out-of-order execution, an instruction-scheduling paradigm that is both difficult to implement correctly and prone to timing side-channel attacks without principled defenses.
Even when software is provably constant-time at the instruction-set architecture (ISA) level, transient execution in out-of-order cores can introduce microarchitectural timing variation that causes leakage.
This thesis contributes a foundation for formally verifying correctness and security properties of hardware circuits, with the end goal of verifying an out-of-order processor.
This thesis proposes a timing-security model for hardware circuits and applies it to simple designs, including a variable-latency multiplier and a sequential multiple-cycle processor.
Using \emph{GoLemma}, a deeply embedded hardware description language in Rocq, we verify functional correctness of crucial out-of-order processor submodules that can later be composed into proofs of larger parent modules.
\end{abstract}

\chapter{Introduction}
This thesis contributes an approach for proving correctness and timing properties for hardware circuits described in the Rocq proof assistant.
This work serves as the first step towards the mechanization of entire correctness and timing proofs for a realistic, modern computer system including a single out-of-order core complete with a memory hierarchy.

\section{Background: out-of-order execution} \label{sec:ooo}
Most modern high-speed processors are based on out-of-order (OoO) execution, which allows the CPU to dynamically reorder instructions based on the availability of functional units (FUs) and input operands, minimizing structural hazards.
Instruction scheduling algorithms supporting out-of-order execution with precise exceptions started with Tomasulo's algorithm~\cite{tomasulo1967}, which first introduced the concept of register renaming to eliminate false dependencies between instructions.
Out-of-order processors combined with another ubiquitous performance optimization, speculative execution, are vulnerable to transient execution attacks, which in-order processors are not.
Principled defenses must be carefully designed to ensure that attackers cannot exploit programs that are safe for in-order processors to execute but unsafe for out-of-order processors.

\Cref{fig:fig1} shows a block diagram for the particular variant of the out-of-order algorithm discussed in this thesis.
The algorithm is a modified design based on the MIPS R10000 (R10K) processor~\cite{r10k1996}.

\begin{figure}[t]
  \centering
  \includegraphics[width=0.75\textwidth]{figures/figure1.png}
  \caption{Diagram for an out-of-order machine based on the R10K processor. Arrows signify communication between submodules.}
  \label{fig:fig1}
\end{figure}

Our algorithm is split into seven separate pipeline stages: fetch (F), decode (D), rename (R), issue (I), execute (X), writeback (W), and retire (C).
The front end fetches (F) instructions from the instruction stream and decodes (D) them into an internal data type the machine can schedule.
Rename (R) then maps architectural registers to a larger physical register file so false dependencies (e.g., WAR and WAW hazards) can be dissolved.
Issue (I) selects ready renamed operations and sends them to functional units that execute the actual computation.
Writeback (W) then posts results to the physical registers so that later instructions can consume operands as they become ready.
Retire (C) retires instructions in program order, committing their effects to the architectural state.
Our design diverges from the R10K algorithm in a two major aspects to simplify which invariants are needed to write proofs.
The most important difference is how the renaming mechanism works.
The original R10K algorithm records, in the reorder buffer (ROB), the previous physical register index for each instruction's destination so that the rename map can be rolled back precisely when a younger path is squashed.
We instead maintain two register alias tables (RATs), avoiding the need to track a per-instruction previous physical destination in the ROB.
The \emph{future} table holds the most up-to-date speculative mapping from architectural registers to physical registers, while the \emph{commit} table records the mapping as of the last retired instruction.
Squashing speculative state can then be performed by reverting the \emph{future} back to the \emph{commit} table.

Rolling back the alias tables alone would be insufficient, because speculative execution can still overwrite physical register contents before those instructions retire; restoring only the \emph{architectural-to-physical} (arch-to-phys) mapping would leave stale or partially updated values visible under the recovered names.
Our design therefore pairs the dual RAT with explicit snapshots of register \emph{values} (not just of the rename map).
Enough snapshots are retained to reconstruct the physical register file as it stood at the last retired instruction, so a squash restores both the mapping from each architectural register to its physical backing and the bit patterns stored in those physical registers at that point.
The maximum number of snapshots is finite, so it bounds the number of branches the machine can speculate ahead; once that budget is exhausted, the issue stage must stall until retire frees snapshot capacity.

A second major deviation concerns speculative memory accesses.
To make timing properties easier to state and prove at first, we delay all speculative loads and stores.
This policy is extremely pessimistic: it sacrifices the performance that comes from speculative execution when interacting with memory and is not meant as a proposal for a serious, production-ready design.
Looking ahead, we expect that a more realistic design can be achieved by pairing speculative memory execution with a dynamic taint-tracking defense, together with a mechanized proof that the resulting design still satisfies the same class of timing-security obligations; that extension is left to future work.

\section{Goal: verifying hardware circuits}
This thesis develops techniques for proving correctness and timing properties of synthesizable hardware described in Rocq.
The intended workflow is to state and prove theorems against source-level circuit semantics, then relate them to the semantics of synthesizable Verilog through a verified compiler.
Prior work on security proofs for hardware tends to reason about simplified ISA-level models instead of RTL.
This thesis focuses on synthesizable circuits because they are consumable by downstream synthesis tools for future evaluation on FPGAs or ASICs.

\section{Correctness} \label{sec:related-correctness}

Several prior projects have tackled hardware correctness by embedding hardware description languages directly into a proof assistant.

Kami~\cite{choi2017} is a hardware description language, embedded in Rocq, for specifying and verifying hardware circuits in the style of Bluespec SystemVerilog.
Circuits are semantically treated as labeled transition systems composed of guarded atomic rules, and correctness arguments proceed by refinement between more-abstract and more-concrete specifications.

Ko\^{i}ka~\cite{koika2020} is another language for rule-based hardware design that gives Bluespec-style rules a precise sequential semantics characterized by the \emph{one-rule-at-a-time} (ORAAT) property.
Circuits are described as collections of rules over register state, and the language provides a scheduler that resolves rule conflicts while preserving a one-rule-at-a-time execution model.
Ko\^{i}ka is accompanied by a verified compiler that lowers circuit descriptions to RTL, providing an end-to-end path from high-level specifications to synthesizable netlists.
However, Ko\^{i}ka does not support a module system, so circuits must be described monolithically and lacks compositional reasoning.

The GoLemma language used in this thesis extends upon Ko\^{i}ka and includes modular refinement reasoning, while foregoing ORAAT semantics and instead handling submodule contention through method duplication.

\section{Security} \label{sec:related-timing}

The Meltdown~\cite{lipp2018} and Spectre~\cite{kocher2019} attacks, disclosed in 2018, demonstrated that speculative execution, while critical for performance, introduces new side-channel vulnerabilities absent in in-order processors.
Both attacks exploit the fact that transiently executed instructions leave measurable traces in microarchitectural state, such as cache-residency patterns, that an attacker can recover.
These attacks motivated a large body of research~\cite{koruyeh2018,maisuradze2018,kiriansky2018,schwarz2019,minkin2019} on transient execution attacks.

Hardware-software contracts for secure speculation~\cite{guarnieri2021} formalize hardware defenses by specifying, for a particular design, both what information is leaked and which speculative paths are explored.
An analysis of an abstract out-of-order processor protected by speculative taint tracking~\cite{yu2019} shows that the defense satisfies a contract in which leakage events are branch targets, addresses of loads and stores, and operands of variable-latency instructions.
The key limitation of this line of work is that it reasons about abstract processor models rather than concrete designs expressible in a hardware description language; this thesis targets the gap to synthesizable RTL.

At the software level, constant-time verification for nondeterministic programs and compilers~\cite{conoly2025} develops techniques for proving constant-time properties in the Rocq proof assistant.
Using a formulation of constant time as a single-copy property over omnisemantics, the work proves end-to-end preservation of constant-time routines compiled from Bedrock2~\cite{erbsen2021} source to RISC-V machine code.
The single-copy formulation of constant time used in this thesis is taken from that work, repurposing the approach from software compilers to hardware verification.

SecSep~\cite{song2025} addresses the gap between source-level constant-time guarantees and speculative execution on real hardware through an assembly-level transformation framework.
SecSep targets processors similar to ProSpeCT~\cite{daniel2023}, which partitions memory into regions whose speculative loads are delayed until non-speculative and regions whose loads are not, provably enforcing the constant-time policy without relying on fencing.
SecSep rewrites compiled assembly programs so that secret and public data are partitioned on the stack, using a new typed assembly language called \emph{Octal} to guide the transformation.
The approach correctly handles secrets on the stack and achieves a low additional runtime overhead on cryptographic programs.

Prior work on hardware timing security~\cite{duxovni2018} builds off the Kami framework to reason about timing-security properties of hardware circuits, showing that the same embedding used to verify functional correctness can also support security arguments.
More recent work~\cite{lau2024} formally specifies and verifies strong timing isolation for hardware enclaves, reasoning about cycle-accurate circuit semantics with respect to an air-gapped-machine formulation of isolation and applying the method to a multicore, pipelined RISC-V design.
This thesis continues in the same tradition, carrying these timing-security proof techniques into the setting of individual submodules of a large out-of-order machine.

\section{State machines} \label{sec:timing-state-machine}

A \emph{state machine} is an abstraction used to describe the behavior of a computation by defining its possible states and the rules for transitioning between them.
Hardware circuits function as physical state machines: registers serve as the memory that holds the current state, while combinational logic defines the functions that determine both the system's outputs and its next state based on the current configuration.
In the context of this thesis, we view hardware modules as interacting through well-defined interfaces composed of methods rather than raw, unstructured signals.
Each discrete state transition in the system corresponds to the execution of an method, supplied with arguments that represent the transaction's data.
This approach is consistent with prior work \cite{choi2017,koika2020} where hardware modules are viewed as high-level transition systems where the transitions lie on method boundaries.
We adopt the following formal definition of a state machine.

\begin{definition}[State machine]
  A state machine $\mathcal{M}$ is a 5-tuple $(S, s_0, I, O, \delta)$ where $S$ is a set of states, $s_0 \in S$ is the initial state, $I$ is a set of inputs, $O$ is a set of outputs, and
  \[
    \delta : S \times I \rightharpoonup S \times O
  \]
  is a partial transition function.
  Here $\xrightharpoonup{}$ denotes that certain methods can abort and produce no state change or output.
\end{definition}

We model hardware circuits as such machines.
A state $s \in S$ captures the current values of the sequential logic relevant to the module.
Each input in $I$ represents a \emph{method invocation}: a variant type carrying a method tag together with the argument values for that call.
The transition function $\delta$ is parital on $S \times I$ to reflect that the model may not produce a well-defined successor state and output when the call is \emph{aborted} by a guard (for example, dequeuing from an empty buffer).
An aborted call leaves the circuit in the same configuration.
This convention simplifies trace definitions and simulation proofs: each input sequence determines one run, and the optional output indicates whether the call produced a visible event in the output trace.

For the purpose of reasoning about security with this framework, we additionally restrict attention to circuits whose behavior is \emph{deterministic} in the sense above: for each $s \in S$ and $i \in I$, there is at most one pair $(s', o) = \delta(s,i)$ with $o \in O$.
This restriction is convenient for reasoning about the proof obligations in \Cref{sec:timing-ct} and \Cref{sec:timing-proof}, where we compare runs under related inputs and want equality of traces without existentially quantifying over implementation-defined resolution of nondeterminism in proofs.
Determinism is a design choice of convenience, not necessity; as discussed in \Cref{chap:golemma}, the framework for deeply embedded circuits can use nondeterministic semantics (in particular, omnisemantics~\cite{chargueraud2023}) to describe specifications.

Given an input sequence $i_1, i_2, \ldots, i_n \in I$, repeatedly applying $\delta$ from the initial state $s_0$ defines a \emph{run} of the machine.
The corresponding sequence of per-step observations is called the machine's \emph{trace}.
Concretely, the machine $\mathcal{M}$ produces a sequence of optional outputs by unfolding $\delta$ from $s_0$:
\[
  s_0 \xrightarrow{i_1 / o_1} s_1 \xrightarrow{i_2 / o_2} s_2 \;\cdots\; \xrightarrow{i_n / o_n} s_n, \qquad o_i \in O,\ s_i \in S.
\]
The \emph{observable trace} of the run is the list $[o_1,\ldots,o_n]$.
For timing security, an adversary interprets elements of $O$ (modulo erasure, described in \Cref{sec:timing-ct}) as the information that is leaked,
Aborted transitions do not produce events in the trace.

The state machine model is mechanized in Rocq as a plain record (\Cref{fig:circuit-typeclass}), parameterized by a state type~\texttt{s}, an input type~\texttt{i}, and an output type~\texttt{o}.
These parameters correspond directly to $S$, $I$, and $O$.

\begin{figure}[t]
\begin{minted}{coq}
Definition Action s t := s -> option (s * t).

Record StateMachine (s i o : Type) := {
  state_machine_rst  : s;
  state_machine_call : i -> Action s o;
}.
\end{minted}
\caption{The \texttt{StateMachine} record. The fields \texttt{state\_machine\_rst}
  and \texttt{state\_machine\_call} correspond to $s_0$ and to the family of
  method bodies that realize $\delta$ when combined with the operational
  semantics of \texttt{Action} (\Cref{sec:timing-monads}).}
\label{fig:circuit-typeclass}
\end{figure}

Method bodies are written in the monad \mintinline{coq}{Action s o}, which can be richly composed together to simulate imperative programs.
The function \mintinline{coq}{step_action} (\Cref{fig:machine-step}) evaluates a body in a given state and normalizes the result to a deterministic Rocq value of type \mintinline{coq}{s * option o}: either the action succeeds and returns an updated state together with \mintinline{coq}{Some out}, or it aborts and returns the original state with \mintinline{coq}{None}.
Composing this with \mintinline{coq}{state_machine_call} yields \mintinline{coq}{machine_step}, the direct analogue of one application of $\delta$.

\begin{figure}[t]
\begin{minted}{coq}
Definition step_action {st o : Type} (a : Action st o) (s : st)
  : st * option o :=
  match run_action a s with
  | Some (s', out) => (s', Some out)
  | None => (s, None)
  end.

Definition machine_step {st i o : Type}
  (sm : StateMachine st i o) (inp : i) (s : st) : st * option o :=
  step_action (state_machine_call sm inp) s.
\end{minted}
\caption{The \mintinline{coq}{step_action} and \mintinline{coq}{machine_step} functions
        describe single-step semantics.}
\label{fig:machine-step}
\end{figure}

Longer runs are defined by structural recursion over input lists (\Cref{fig:machine-runs}).
The helper \mintinline{coq}{run_machine} applies \mintinline{coq}{machine_runs} from the reset state \mintinline{coq}{state_machine_rst sm} and produces a trace of optional outputs in reverse order.

\begin{figure}[t]
\begin{minted}{coq}
Fixpoint machine_runs {st i o : Type}
  (sm : StateMachine st i o) (s : st) (inputs : list i) : st * list (option o) :=
  match inputs with
  | [] => (s, [])
  | inp :: inputs' =>
      let '(s', out) := machine_step sm inp s in
      let '(s'', outs) := machine_runs sm s' inputs' in
      (s'', out :: outs)
  end.

Definition run_machine {st i o : Type}
  (sm : StateMachine st i o) (inputs : list i) : st * list (option o) :=
  machine_runs sm (state_machine_rst sm) inputs.
\end{minted}
\caption{The function \texttt{machine\_runs} executes a \texttt{StateMachine}
  on a list of inputs, threading state forward and producing one optional output
  per input.}
\label{fig:machine-runs}
\end{figure}

\section{Slate: encoding circuits with guarded state monads} \label{sec:timing-monads}

A \emph{shallow embedding} encodes an object language directly as terms of the host language.
In this style, the host's type system, reduction rules, and terms \emph{are} the semantics of the embedded language.
In Rocq, that means method bodies are ordinary Gallina functions rather than abstract syntax trees requiring a separate interpreter.
By contrast, a \emph{deep embedding} defines an explicit syntax by a series of inductive datatypes representing commands and expressions of the language. A deep embedding must come with a separate interpreter or relationally-defined operational semantics.

Each style has distinct advantages and costs.
Shallow embeddings are typically easier to do verification work with, because you can easily leverage the facilities Rocq has for Gallina already; many proof steps reduce by unfolding and simplification rather than having to reason about an interpreter or operational semantics.
The main drawback is that source-level program transformations, such as compiler passes, are less direct, since syntax is not explicit; these tasks often require reification and tactic support to recover a manipulable syntax tree.

Deep embeddings make syntax explicit from the beginning, so compiler construction and optimization passes are easier to express.
However, they impose additional verification overhead: one must define and reason about an interpreter (or step relation), and prove that this semantics aligns with the intended meaning.
As a result, many proofs are more involved for similar theorems.

\paragraph{State monads}
A \emph{state monad} over state type $s$ and result type $t$ is defined as
\[
  \texttt{State}\;s\;t \;\triangleq\; s \to (s * t)
,\]
that is, a function that takes an input state and returns a new state together with a result.
A stateful computation can be understood as a function that consumes a state and returns a pair of the updated state and a result value.
This representation is powerful because it allows you to express side effects sequentially by chaining together monads using \mintinline{coq}{bind}.
Mutable references do not need to be primitively supported by the meta-language (i.e. Gallina); instead, they are ordinary functions that transform an explicit state argument.
As a consequence, one can describe sequential hardware behavior in a pure functional setting and still recover the usual intuition of sequential logic, by reading registers at the start of the cycle, performing some combinational function on those values, and updating the registers at the end of the cycle.

\Cref{fig:state-monad-core} shows the state monad encoded in Rocq together with its basic operations.
The constructor \mintinline{coq}{ret} injects a pure value into the monad without changing the state, while \mintinline{coq}{bind} sequences two computations by feeding the result and post-state of the first into the second.
The infix operator \mintinline{coq}{>>=} is another way to express \mintinline{coq}{bind}.
State monads also provide state-specific primitives, notably \mintinline{coq}{get} and \mintinline{coq}{put}: \mintinline{coq}{get} returns the current state as a value, and \mintinline{coq}{put} replaces the current state with a new one.

\begin{figure}[t]
\begin{minted}{coq}
Definition State (s t : Type) :=
  s -> (s * t).

Definition ret {s t} (x : t) : State s t :=
  fun st => (st, x).

Definition bind {s t1 t2} (m : State s t1) (f : t1 -> State s t2) :=
  fun st =>
    let '(st', x) := m st in
    f x st'.

Definition get {s} : State s s :=
  fun st => (st, st).

Definition put {s} (x : s) : Action s unit :=
  fun _ => (x, tt).
\end{minted}
\caption{State monad definitions in Rocq.}
\label{fig:state-monad-core}
\end{figure}

\paragraph{Guarded state monads}
A \emph{guarded state monad}, also referred to as an \emph{action}, over state type $s$ and result type $t$ is the type of computations
\[
  \texttt{Action}\;s\;t \;\triangleq\; s \to \texttt{option}\;(s * t)
,\]
that is, a function that takes an input state and either returns a new state together with a result (\mintinline{coq}{Some}) or aborts (\mintinline{coq}{None}).
The framework uses this guarded form to model method calls that are enabled only when their guards hold.
The \mintinline{coq}{Some} case represents a successful step that returns an updated state and a value.
The \mintinline{coq}{None} case represents an aborted step (for example, when a guard is false).
This extra layer is exactly what is needed for hardware method interfaces where not all calls are enabled in all states.

\Cref{fig:guarded-state-monad-core} shows the guarded state monad encoded in Rocq together with its basic operations. \texttt{ret}, \texttt{bind}, \texttt{get}, and \texttt{put} work similarly to their state monad analogues; however, they are lifted into option types by using the \texttt{Some} constructor.

\begin{figure}[t]
\begin{minted}{coq}
Definition Action (s : Type) (t : Type) :=
  s -> option (s * t).

Definition ret {s t} (x : t) : Action s t :=
  fun st => Some (st, x).

Definition bind {s t1 t2} (m : Action s t1) (f : t1 -> Action s t2) :=
  fun r =>
    match m r with
    | Some (s, x) => f x s
    | None => None
    end.

Definition get {s} : Action s s :=
  fun st => Some (st, st).

Definition put {s} (x : s) : Action s unit :=
  fun _ => Some (x, tt).

Definition modify {s} (f : s -> s) : Action s unit :=
  fun st => Some (f st, tt).

Definition fail {s t} : Action s t :=
  fun _ => None.
\end{minted}
\caption{Guarded state monad definitions in Rocq.}
\label{fig:guarded-state-monad-core}
\end{figure}

\paragraph{Modify}
\mintinline{coq}{modify f} updates the state by applying a function \mintinline{coq}{f}.
It can be read as a derived write primitive: read the current state, transform it, then commit the result.
In hardware, this operation is a natural fit for register updates where the next state is expressed as a pure function of the current state.
Proof obligations involving \mintinline{coq}{modify} usually reduce to equalities between functions applied to the same input state.

\paragraph{Guard failure}
\mintinline{coq}{fail} returns \mintinline{coq}{None} for any input state.
In hardware description semantics, this represents an aborted method call (for example, violating a ready condition).
At the state machine level, this is later normalized into a step with no observable output and no state changes.
Introducing failure in the monad itself keeps reasoning about guards local to method definitions.

\paragraph{Submodule calls}
The operator \mintinline{coq}{call} implements modular method calls over nested state.
It uses \mintinline{coq}{getter} to project a substate, runs an \mintinline{coq}{Action} on that substate, and then re-integrates the updated substate using \mintinline{coq}{setter} as shown in \Cref{fig:action-call}.
If the callee action fails, the caller fails.
This mirrors calling semantics in Bluespec SystemVerilog, where a method call failing to fire causes the caller to not fire.

\begin{figure}[t]
\begin{minted}{coq}
Definition call {s a b}
  (getter : s -> a)
  (setter : a -> s -> s)
  (action : Action a b) : Action s b :=
  fun st =>
    fun _ =>
      match action (getter st) with
      | Some (st', x) => Some (setter st' st, x)
      | None => None
      end).
\end{minted}
\caption{The \texttt{call} operator lifts the effects of submodule action to a larger state.}
\label{fig:action-call}
\end{figure}

\paragraph{Syntax}
The syntax for the surface language, \emph{Slate} (surface language for actions), used in method bodies is implemented using the notation facilities in Rocq.
Crucially, this notation elaborates into the same Gallina terms defined above; it does not introduce a syntax tree.
Slate represents a shallow embedding.
An abstract syntax of the grammar is given in \Cref{fig:action-grammar}; the concrete Rocq notation declarations are listed in \Cref{appendix:action-notation}.
The terminal \texttt{term} in the grammar refers to arbitrary Gallina terms.
To properly embed a Gallina term as a statement, it must typecheck to an \mintinline{coq}|Action s t| where \mintinline{coq}{s} matches the state type of the surrounding context.

\begin{figure}[t]
\[
\begin{array}{rcl}
\text{Expressions} \quad e & ::= & e\ \mathit{==}\ e \mid \mathit{term} \mid \texttt{`}\mathit{reg}\texttt{`read()} \\
\\
\text{Statements} \quad s & ::= & \texttt{pass} \mid \texttt{abort} \mid \texttt{return}\ e \mid \texttt{`}\mathit{reg}\texttt{`write}(e) \\
& & \mid \texttt{if}\ (e)\ s\ \texttt{else}\ s \mid \texttt{when}\ (e)\ s \mid \texttt{\{\{}~\mathit{term}~\texttt{\}\}} \\ & & \mid \texttt{let}\ \mathit{pat}\ \texttt{=}\ e \mid \texttt{begin}\ b\ \texttt{end} \mid \texttt{call}\ \texttt{`}\mathit{submodule}\texttt{`}(e) \\
\\
\text{Blocks} \quad b & ::= & s \mid s\ \texttt{;}\ b \\
\\
\text{Actions} \quad a & ::= & \texttt{begin}\ b\ \texttt{end}
\end{array}
\]
\caption{Abstract syntax of the surface language for actions.}
\label{fig:action-grammar}
\end{figure}

\paragraph{Example}
Consider \Cref{fig:empty-surface} which implements the body of a method called \mintinline{coq}{mark_busy} which operates over the state type \mintinline{coq}{State}.
In the surface language, the method reads the phase register into the let binding \mintinline{coq}{ph}, checks the condition \mintinline{coq}{ph == Empty}, writes to the phase register, and returns \mintinline{coq}{OutEnq}.
This style is close to imperative pseudocode while still denoting a pure Gallina term.
The same method can be written directly with monadic operators (\Cref{fig:empty-bind}).
This version makes sequencing explicit through \mintinline{coq}{>>=}, but is less readable for longer method bodies.
In practice, both forms are definitionally equivalent after notation expansion.


\begin{figure}[t]
\begin{minted}{coq}
Variant Phase := Busy | Empty | Ready.

Definition State := { phase: Phase }.

Definition mark_busy : Action State bool :=
  begin
    let ph = `phase`read();
    if (ph == Empty) begin
      `phase`write(Busy);
      return true
    end else begin
      return false
    end
  end.
\end{minted}
\caption{Method \mintinline{coq}{mark_busy} using the surface language for actions.}
\label{fig:empty-surface}
\end{figure}

\begin{figure}[t]
\begin{minted}{coq}
Definition mark_busy : Action State bool :=
  read phase >>= fun ph =>
  if (ph == Empty) then
    modify (fun s => s <| phase := Busy |>) >>= fun _ =>
    ret true
  else
    ret false.
\end{minted}
\caption{Method \mintinline{coq}{mark_busy} using inline monadic notation.}
\label{fig:empty-bind}
\end{figure}

Proof scripts over guarded state monads require automation to avoid manually $\delta$--reducing or unfolding monadic operators (for example, \mintinline{coq}{bind} or \mintinline{coq}{ret}) all the time.
Goals are also often blocked until helper definitions are unfolded (for example, \mintinline{coq}{run_action}, \mintinline{coq}{machine_step}, and \mintinline{coq}{step_action}).
Accordingly, the development style combines two mechanisms.
First, specialized simplification tactics perform local reductions.
Second, hint databases (i.e. \mintinline{coq}{Hint Unfold}) provide controlled automation so the same normalization tactics do not need to be applied for each lemma.

\section{Security definitions} \label{sec:timing-ct}

Cryptographic implementations are often validated against the \emph{cryptographic constant time} (CCT) property: attackers should not observe leakage events such as branch direction, memory-access patterns, or variable-latency instructions influenced by secret inputs.
This principle has been formalized and mechanically verified for software programs in mulitple prior works~\cite{almeida2016,conoly2025,guarnieri2021}.
\Cref{fig:cct-intuition} illustrates the key intuition with two small routines: one has a secret-dependent branch and is not constant-time, while the other computes through bit-masking and avoids control-flow dependent on secret data.

More formally, constant-time security is stated as an observational equality under varying secrets: if two executions agree on public inputs, then they must be indistinguishable with respect to the attacker-observable trace.
The next definition makes this precise in the traditional two-copy (hyperproperty) style, before introducing an equivalent one-copy formulation.

\begin{figure}[t]
\begin{minipage}[t]{0.48\linewidth}
\begin{minted}{c}
// Not constant-time
uint32_t sel_bad(x, y, secret) {
  uint32_t x;
  uint32_t y;
  uint8_t secret;

  if (secret) return x;
  else return y;
}
\end{minted}
\end{minipage}\hfill
\begin{minipage}[t]{0.48\linewidth}
\begin{minted}{c}
// Constant-time
uint32_t sel_ct(x, y, secret) {
  uint32_t x;
  uint32_t y;
  uint8_t secret;

  uint32_t m = 0u - (secret & 1u);
  return (x & m) | (y & ~m);
}
\end{minted}
\end{minipage}
\caption{Constant time example: secret-dependent branch (left) versus branch-free masked selection (right).}
\label{fig:cct-intuition}
\end{figure}

\begin{definition}[Cryptographic constant time as a two-copy hyperproperty] \label{def:two-copy}
A state machine $\mathcal{M} = (S, s_0, I, O, \delta)$ is \emph{cryptographic constant time} (CCT)
if there exist partitions
\[
  s = (s^{\mathrm{pub}}, s^{\mathrm{sec}}), \quad
  i = (i^{\mathrm{pub}}, i^{\mathrm{sec}}), \quad
  o = (o^{\mathrm{pub}}, o^{\mathrm{sec}})
\]
of states, inputs, and outputs into public and secret components, such that for any two
states $s_1, s_2 \in S$ with $s_1^{\mathrm{pub}} = s_2^{\mathrm{pub}}$, and for any two
input sequences $(i_1, i_2, \ldots)$ and $(i_1', i_2', \ldots)$ in $I$ with $i_k^{\mathrm{pub}} = i_k^{\prime\,\mathrm{pub}},$
the public output components agree at every step $k$, $o_k^{\mathrm{pub}} = o_k^{\prime\,\mathrm{pub}}.$
\end{definition}

This definition is relational: it compares two universally-quantified executions and is therefore a hyperproperty.
Next, we introduce a one-copy formalism of constant time which is logically equivalent to \Cref{def:two-copy}.

\begin{definition}[Cryptographic constant time as a one-copy property]
A state machine $\mathcal{M} = (S, s_0, I, O, \delta)$ is \emph{cryptographic constant time} (CCT)
if given partitions of states, inputs, and outputs
\[
  s = (s^{\mathrm{pub}}, s^{\mathrm{sec}}), \quad
  i = (i^{\mathrm{pub}}, i^{\mathrm{sec}}), \quad
  o = (o^{\mathrm{pub}}, o^{\mathrm{sec}})
\]
there exists a predictor function $f_\textsf{predict} : S^{\mathrm{pub}} \times (I^{\mathrm{pub}})^* \to (O^{\mathrm{pub}})^*$
where $\mathsf{pub\_inputs}(\vec{i})$ is the elementwise projection of an input sequence
to public components and $\mathsf{pub\_output}(\vec{o})$ is the elementwise projection of the output trace to its public components.
Then, for every initial state $s_0 \in S$ and input sequence $\vec{i} \in I^*$,
\[
  \mathsf{pub\_outputs}(M,s_0,\vec{i})
  =
  f_\textsf{predict}\bigl(s_0^{\mathrm{pub}},\mathsf{pub\_inputs}(\vec{i})\bigr).
\]
\end{definition}

This definition originally comes from work on retrofitting the Bedrock2 compiler to prove preservation of cryptographic constant time across compiler passes~\cite{conoly2025}.
Intuitively, the observable behavior is fully determined by the public projection of the input.
This is a one-copy specification style: each run is characterized independently without needing to relate two separate sequences.
This definition is useful for certain proof techniques that can existentially construct a nice predictor function ($f_\textsf{predict}$).
The hardware development in this thesis uses the same one-copy perspective, but instantiated to state-machine traces with explicit input/output erasure functions that capture the notion of the secret/public partitions.
We now give the hardware constant-time definition in the style used by the formalization.

\newpage

\begin{definition}[Hardware constant time]
A state machine $\mathcal{M}=(S,s_0,I,O,\delta)$ is \emph{hardware constant time} (HCT) with respect to
\emph{erasure specification}
$\mathcal{E} = (\mathsf{erase_{in}}, \mathsf{erase_{out}})$
where
$\mathsf{erase_{in}} : I \to I$ replaces secret input components with default values and
$\mathsf{erase_{out}} : O \xrightharpoonup{} O$ projects outputs to their public components.

Given an input sequence $\vec{i} \in I^*$, the \emph{erased trace} of $\mathcal{M}$ is defined as
\[
  \mathsf{erase\_trace}(\mathcal{M}, \mathcal{E}, \vec{i})
  \;\triangleq\;
  \mathsf{map}\;\mathsf{erase_{out}}\;\bigl(\mathsf{trace}(M, s_0, \vec{i})\bigr).
\]
Then $M$ is \emph{hardware constant-time} (HCT) with respect to $\mathsf{spec}$ iff
\[
  \forall \vec{i} \in I^*.\;\;
  \mathsf{erase\_trace}(M, \mathsf{spec}, \vec{i})
  \;=\;
  \mathsf{erase\_trace}\!\bigl(M, \mathsf{spec}, \mathsf{map}\;\mathsf{erase_{in}}\;\vec{i}\bigr).
\]
That is, erasing secrets from the inputs before running the machine does not change the erased output trace.
\end{definition}

Note that the definition compares two runs of the \emph{same} machine $M$: one on the original inputs $\vec{i}$ and one on the erased inputs $\mathsf{map}\;\mathsf{erase_{in}}\;\vec{i}$.
This is a self-refinement property and corresponds directly to the Rocq definition:

\begin{minted}{coq}
Definition hardware_constant_time {st i o : Type}
  (sm : StateMachine st i o) (spec : ErasureSpec i o) : Prop :=
  outputs_erased_equivalent sm sm spec.
\end{minted}
